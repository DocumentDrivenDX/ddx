HELIX is the opinionated workflow that runs on top of DDx — phases, gates, and bounded actions for teams that want a methodology out of the box.

The problem

AI agents can write code. They cannot hold context across a codebase, a sprint, or a team.

Without structure, agents drift — regenerating work, ignoring dependencies, reverting to bad states. DDx gives agents a contract to execute against: acceptance criteria on every task, specs that encode intent, and evidence that makes every run inspectable.

How it works

01

Define the work

Each task declares acceptance criteria and its dependencies upfront. Nothing runs without a contract.

02

Execute in isolation

Agents run in dedicated git worktrees — no shared state, no context bleed. Results merge cleanly or are preserved for review.

03

Review against criteria

Every run is graded against its acceptance criteria. Work that passes closes. Work that fails re-queues with the findings attached.

Principles

01

Platform, Not Methodology

DDx ships primitives — beads, agent dispatch, document library. Workflow opinions live in plugins like HELIX, never in the core.

02

Project-Local by Default

Every plugin, persona, and bead lives under the project root. No global state, no machine-wide side effects, no surprises.

03

Validate Your Work

Every bead carries acceptance criteria. Every agent run leaves evidence. Reviews gate close — failing beads re-queue with findings.

04

Evidence Over Promises

Each agent invocation writes inputs, outputs, and cost into the execution bundle. You can always tell which model produced which diff.

05

Cost-Tiered Routing

Cheap models do routine work, strong models review, deterministic checks sit at the top of the ladder. Optimize closed-work-per-dollar.

06

Shared Library, Not Silos

Patterns, prompts, and personas flow back to the master library via git subtree. What works in one repo reaches the next.
