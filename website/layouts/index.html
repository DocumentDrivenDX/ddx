Document-driven development

Documents drive the agents. DDx drives the documents.

DDx is the local-first platform for the documents AI agents consume to build software — beads carry work, specs carry intent, plugins package workflows, and every agent run leaves evidence behind.

Pairs with HELIX

DDx is the platform. HELIX is the opinionated workflow that runs on top of it — phases, gates, and bounded actions for teams that want a methodology out of the box.

Work moves when the load, lever, and fulcrum are visible. DDx operating principle

The new reality

Agents now write the code. The bottleneck moved upstream — to the documents that tell them what to build.

Specs, beads, prompts, personas, and review notes are the new source code. They need a workspace, a tracker, an audit trail, and a way to travel between projects. DDx is that workspace.

What breaks without it

01

Prompts scatter across machines

Every developer keeps their own folder of half-edited instructions. None of them ship with the project.

02

Specs drift from reality

The spec said one thing, the code did another, and nobody updated the doc the agent reads tomorrow.

03

Work has no shared queue

Tickets live in one tool, AI tasks in another, and neither knows which agent is allowed to pick them up.

04

Agent runs leave no evidence

You cannot tell which prompt, which model, or which inputs produced the diff that just landed.

05

Strong models do cheap work

Routine edits burn frontier tokens because there is no ladder for which model handles which class of task.

06

Improvements stay trapped

A pattern that works in one repo never reaches the next, because nothing shares it back to a library.

The physics

  • Load the work — beads with acceptance and dependencies.
  • Lever the agents — prompts, personas, and harnesses you reuse.
  • Fulcrum the evidence — every run, in the repo, reviewable.
