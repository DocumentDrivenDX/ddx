# Plan: DDx Full-Spectrum Platform Stabilization

Date: 2026-04-04
Scope: Repo-wide
Status: Draft for implementation planning

Note: execution and runtime-storage details in this draft follow the newer
bead-backed collection model from `ADR-004`, `TD-010`, `TD-005`, and `TD-006`.

This document captures the complete implementation plan for the DDx repository against the HELIX planning stack. It addresses active FEAT-001 through FEAT-009 work, current implementation gaps, and required design-level decisions before broad execution.

## 1) Problem Statement and User Impact

DDx defines itself as a document-driven infrastructure for AI-assisted development, but the repo currently has a split maturity profile:

- Core CLI commands for documents, beads, agent dispatch, and doc graph logic are mostly in place.
- Server capabilities are partially implemented and do not satisfy the full FEAT-002/FEAT-008 contract.
- FEAT-009 registry workflows are unimplemented despite planning and open execution items.
- Some planning artifacts were previously inconsistent with `ddx:` governance conventions.

### What this plan enables

If executed as written, DDx moves from “partially implemented and internally useful” to a coherent, production-ready platform where:

- all user-facing features are covered by contracts and tests,
- artifact governance is internally consistent,
- service surfaces (CLI/server/MCP/web) are aligned,
- and future features can be shipped from a documented, bounded architecture.

Without this plan, the repo risks:

- silent drift between feature specs and runtime behavior,
- repeated reimplementation of interfaces in CLI/server/web,
- and security exposure from unauthenticated service endpoints.

## 2) Requirements Analysis

### 2.1 Functional requirements (must-have)

#### 2.1.1 CLI

- Keep implemented command families stable:
  - Beads: `ddx bead` CRUD + deps + ready + status.
  - Agents: `ddx agent run/list/doctor/log`.
  - Doc graph: `ddx doc graph/stale/stamp/show/deps/dependents/validate`.
  - Server: `ddx server` startup with configurable bind address and port.

- Implement missing FEAT-002 and FEAT-008 capabilities:
  - Full server API surface: 
    - docs: list/read/search + persona resolution + graph/deps APIs,
    - beads: list/show/ready/blocked/status + dep tree,
    - doc graph: graph/stale/docs/deps/dependents,
    - agent sessions: list/detail and optional filtering,
    - health/readiness/info endpoints.
  - MCP tools corresponding to all server APIs.
  - Static web UI embedding via `embed.FS` and SPA fallback.

- Implement FEAT-009 end-to-end:
  - `ddx update`, `search`, `install`, `installed`, `uninstall` for resources and workflow packages.
  - `.ddx/lock.yaml` with registry/repo/commit/tree hashes and per-file hashes.
  - `ddx verify` to validate lock integrity offline.

- Complete remaining FEAT-007 parity gap:
  - `ddx doc migrate` conversion from legacy `dun:` frontmatter to `ddx:`.

#### 2.1.2 Agent service

- Wire `ddx agent` config reads from `.ddx/config.yaml` fields and env override order.
- Preserve CLI defaults when flags are set.
- Surface session path and metadata consistently across `agent.log` and server APIs.

#### 2.1.3 Configuration and observability

- Add explicit server configuration model for:
  - bind address/port,
  - optional API key (default off),
  - CORS allowed hosts,
  - log level and telemetry settings.

### 2.2 Non-functional requirements

- Performance:
  - Command responses under 200 ms for local filesystem/read-only operations.
  - `ddx doc graph` and `ddx docs stale` complete under 500 ms for ~100+ markdown files.
  - `ddx-server` document read endpoints under 200 ms, graph/stale under 500 ms on warm cache.

- Reliability:
  - Deterministic doc graph hashing, explicit staleness reasons, and stable output ordering.
  - No data loss in bead/lock writes under concurrent access.
  - Fail-safe behavior: degrade to best-effort mode with clear user-facing errors.

- Operability:
  - Minimal dependencies for execution.
  - Structured logs with request path, status, duration, and correlation identifiers.

### 2.3 Constraints and assumptions

- Multi-file or remote DB backends are out of scope in v1; source remains file-backed.
- Localhost default for server, optional auth for remote use.
- Use established ADR stack:
  - Go + slog + OpenTelemetry (`ADR-001`),
  - Bun + Vite + React + Biome + Vitest/Playwright (`ADR-002`),
  - Lockfile + tree hash integrity (`ADR-003`).

### 2.4 Open questions to resolve before implementation

1. Should FEAT-002 enforce API key on all mutating endpoints only, or all endpoints uniformly?
2. For web UI, should BE-level pagination be mandatory from day 1 (future large repos) or deferred?
3. Registry verify flow for first-fetch trust: should ADR-003 “first-fetch TOFU” be surfaced interactively, or silent with warning?

## 3) Architecture Decisions

### Decision 3.1: Keep server architecture single-binary, single-process with three surfaces

Use one `ddx-server` process with three surfaces:

- `/api/` REST for UI/automation;
- `/mcp/` Streamable HTTP-compatible JSON-RPC tool interface;
- SPA root `/` from embedded assets.

Rationale:
- Existing implementation already uses this model in `cli/cmd/server.go` and `cli/internal/server/server.go`.
- Maintains deployment simplicity and avoids sidecar complexity.

Alternatives considered:
- Separate server + UI binary pair: reintroduces deployment coupling.
- Pure CLI-only only with no server/web: blocks FEAT-002/FEAT-008 and web adoption.

### Decision 3.2: Treat docgraph package as core platform engine

Promote docgraph to shared read model for:
- CLI (`ddx doc ...`),
- server graph endpoints and MCP graph tools,
- future UI graph visualizers.

Rationale:
- Already implemented with deterministic hash, dependency graph, cascade, and write-back via `frontmatter`.
- Existing issue `ddx-98dae668` is in progress and should own graph integrity.

### Decision 3.3: Preserve existing bead storage but standardize API responses

Continue JSONL as default bead backend while standardizing server/CLI serialization shape for all endpoints.

Rationale:
- `bead` implementation is stable and tested.
- Avoids breaking CLI and existing workflow storage workflows.

### Decision 3.4: Introduce explicit API contract envelope layer in server

Introduce conversion layer for all server endpoints:
- normalize validation,
- fixed error payload shape,
- stable field ordering,
- JSON schema compatibility checks.

Rationale:
- Current server handlers return mixed payload/error patterns.
- FEAT-002/007/008 and UI integration need stable contracts.

### Decision 3.5: Build registry package with pluggable remote fetch providers

Registry fetching strategy:
- `registry.yaml` index fetch,
- package descriptor resolution,
- deterministic tar/clone fetch,
- install verification via tree and per-file hash.

Rationale:
- Supports FEAT-009 package model and lockfile integrity from ADR-003.

## 4) Interface Contracts

### 4.1 CLI contracts (selected)

#### 4.1.1 New/updated commands and behaviors

- `ddx server [--addr --port]`
  - starts HTTP+MCP+
  - reads effective config from flags, env, and `.ddx/config.yaml`.

- `ddx doc migrate [--all|<path>]` (if scoped)
  - converts `dun:` to `ddx:` while preserving unknown fields,
  - fails on malformed frontmatter with actionable error.

- `ddx doc show <id> --json`
  - must include hash + stale reasons + dependents.

- `ddx update`, `search`, `install`, `installed`, `uninstall`, `verify`
  - all return machine-readable JSON with `--json`.

#### 4.1.2 Agent config contract

`agent` command behavior:

1. `ddx agent run` resolves in order:
   1. explicit flags,
   2. `.ddx/config.yaml` section `agent.*`,
   3. env defaults,
   4. built-in defaults.
2. `agent.log` reads configured `agent.session_log_dir`.
3. `agent.doctor` and `agent.list` remain no network-bound and deterministic.

### 4.2 HTTP API contracts (v1)

All request/response are JSON.

#### 4.2.1 Core response style

```json
{
  "data": "...",
  "meta": {
    "generated_at": "RFC3339",
    "request_id": "uuid",
    "path": "/api/xxx"
  }
}
```

Error shape:

```json
{
  "error": {
    "code": "ERR_INVALID_REQUEST",
    "message": "...",
    "details": {"field": "..."}
  },
  "meta": {...}
}
```

#### 4.2.2 Documents

- `GET /api/documents`
  - query: `type`, `q`, `updated_after`, `root`, `limit`, `offset`.
  - response: list of document entries with `path`, `category`, `updated_at`, `has_ddx_frontmatter`, `id`.

- `GET /api/documents/:path`
  - path traversal blocked; URL-encoded relative path expected.

- `GET /api/search?q=`
  - full-text + frontmatter search.

- `GET /api/personas/:role`
  - resolves role via `.ddx/config.yaml` persona bindings.

#### 4.2.3 Beads

- `GET /api/beads` supports `status`, `label`, `limit`, `offset`.
- `GET /api/beads/:id`
- `GET /api/beads/ready`
- `GET /api/beads/blocked`
- `GET /api/beads/status`
- `GET /api/beads/dep/tree/:id`

#### 4.2.4 Doc graph

- `GET /api/docs/graph`
- `GET /api/docs/stale`
- `GET /api/docs/:id`
- `GET /api/docs/:id/deps`
- `GET /api/docs/:id/dependents`
- `GET /api/docs/:id/history` (optional future v1.1).

#### 4.2.5 Agent logs

- `GET /api/agent/sessions`
  - query: `limit`, `harness`, `status`, `since`.
- `GET /api/agent/sessions/:id`

#### 4.2.6 Registry (FEAT-009)

- `GET /api/registry/search?q=`
- `GET /api/registry/installed`
- `POST /api/registry/refresh`
- `POST /api/registry/install`
- `POST /api/registry/uninstall`
- `GET /api/registry/verify`

#### 4.2.7 Health and diagnostics

- `GET /api/health` (liveness)
- `GET /api/ready` (dependency checks)
- `GET /api/metrics` (optional if OTEL exporter active)

### 4.3 MCP contracts

Supported tool naming and behavior should map to existing FEAT-002 list:

- `ddx_list_documents`
- `ddx_read_document`
- `ddx_search`
- `ddx_resolve_persona`
- `ddx_list_beads`
- `ddx_show_bead`
- `ddx_bead_ready`
- `ddx_bead_status`
- `ddx_doc_graph`
- `ddx_doc_stale`
- `ddx_doc_show`
- `ddx_doc_deps`
- `ddx_agent_sessions`

Each tool must return stable JSON serialized content where arrays are sorted deterministically.

### 4.4 UI contract boundaries

- `/` serves SPA index, with fallback routing.
- SPA consumes:
  - list+detail endpoints,
  - health/status,
  - mutation endpoints for configuration only (if introduced).
- Write operations should only run through explicit server APIs with CSRF-aware session boundaries if user-auth mode is added.

## 5) Data Model

### 5.1 Repository canonical data

1. **Artifacts (`*.md`)**
   - `ddx:` frontmatter required for graph inclusion.
   - fields: `id`, `depends_on`, `inputs`, `review`, `prompt`, `parking_lot`.

2. **Bead-backed runtime collections (`.ddx/*.jsonl` for JSONL backend)**
   - the active tracker uses the `beads` collection, which maps to
     `.ddx/beads.jsonl` in the JSONL backend.
   - runtime services use separate named collections such as
     `exec-definitions`, `exec-runs`, and `agent-sessions`.
   - the shared bead envelope stays portable across JSONL and external
     backends, with domain-specific data in preserved extras.

3. **Agent config (`.ddx/config.yaml`)**
   - currently minimal schema in `cli/internal/config`.
   - requires expansion for `agent` and server settings while preserving backward compatibility.

4. **Agent sessions (`agent-sessions` collection + attachments)**
   - for the JSONL backend, session metadata maps to
     `<session_log_dir>/agent-sessions.jsonl`, where `session_log_dir`
     defaults to `.ddx/agent-logs`.
   - prompt, response, stdout, stderr, and correlation data live in sidecar
     attachments under `<session_log_dir>/agent-sessions.d/`.

5. **Execution definitions and runs (`exec-definitions` / `exec-runs`)**
   - for the JSONL backend, metadata maps to `.ddx/exec-definitions.jsonl` and
     `.ddx/exec-runs.jsonl`.
   - large result and log bodies live in sidecar attachments under
     `.ddx/exec-runs.d/`.

6. **Registry index cache (`.cache` or `.ddx/.cache`)**
   - registry metadata + package descriptors + package tarball snapshot hashes.

7. **Lockfile (`.ddx/lock.yaml`)**
   - `packages[*].repo`, `commit`, `tree_hash`, optional `files[*].hash`, install timestamps.

### 5.2 Lifecycle/ownership

- CLI writes:
  - docs via docs commands,
  - beads via bead store,
  - logs via agent runner,
  - lockfile on install/verify.
- Server exposes read/write APIs for selected resources.
- External write should be server-mediated once web UI write features are enabled in v1.1.

### 5.3 Integrity constraints

- IDs are globally unique in each project bead dataset.
- Graph nodes require non-empty `id`.
- Lockfile must be version controlled and reviewed on drift.

## 6) Error Handling Strategy

### 6.1 Input and validation errors

- 400 for malformed JSON/path/flags.
- 404 for missing resources.
- 409 for conflict (e.g., duplicate install in progress).
- 422 for schema validation failures.
- 500 for local file/system/service faults.

All errors must include stable codes and hints.

### 6.2 Fallback and retries

- Registry operations use retry with backoff on transient network errors.
- Bead lock acquisition: wait up to configurable timeout; return clear timeout message with remediation.
- Graph scanning failures in one file: warning collection + continue; treat overall command as success with warnings.

### 6.3 Failure blast radius

- If MCP tool fails, return tool-level error payload, not process crash.
- If one endpoint fails, keep other endpoints operational.
- Server startup should fail fast if config directory unreadable or invalid auth mode selected, with explicit log reason.

### 6.4 Malformed inputs and malformed frontmatter

- `ddx doc validate` must emit all warnings.
- `ddx doc graph` should continue with warnings and exclude invalid/unreadable docs from graph nodes.

### 6.5 Fallback behavior

- Missing library path: return empty list for document browsing endpoints, but `/health` remains available.
- Missing `.ddx/config.yaml`: defaults apply.

## 7) Security Considerations

### 7.1 Threat model

- Unauthorized filesystem read/write through `/api/documents/:path` and path traversal.
- Open server on public network exposing local file and session metadata.
- Integrity risks in lockfile and package content.
- Token leakage from session logs.

### 7.2 Mitigations

1. Path sanitization and canonicalization in all file-based handlers.
2. API key support for non-localhost:
   - default none on localhost,
   - optional header `Authorization: Bearer ...` for remote.
3. CORS restricted to configured origins.
4. Structured secrets handling:
   - never log raw tokens,
   - redact long token-like strings from logs.
5. Lockfile verification and tree-hash checks before install/verify.
6. Signed release advisories optional in future ADR extension.

### 7.3 Server security modes

- Dev mode: localhost only, no key required.
- Remote mode: API key required by default for mutation and sensitive read endpoints.

### 7.4 Auditability

- Record actor/request id for writes and install operations.
- Store immutable event entries for package installs/verifications.

## 8) Test Strategy

### 8.1 Unit tests

- Parser and hashing: expand `cli/internal/docgraph` tests (root path resolution, malformed frontmatter, review block canonicalization).
- Agent runner: preserve existing table tests and add config-binding tests.
- Registry package: deterministic hash/path behavior.

### 8.2 Integration tests

- CLI acceptance for every contract under FEAT-002/007/009.
- Server HTTP integration tests by endpoint family:
  - documents, beads, docs graph, agent sessions, registry flows.
- MCP contract tests for tool discovery and tool call schemas.
- Authenticated endpoint smoke tests for remote mode.

### 8.3 End-to-end tests

- Launch `ddx server` + SPA and execute:
  - doc graph render + stale highlight,
  - bead queue views,
  - agent session view,
  - registry list/install simulation with local mock registry endpoint.

### 8.4 Fuzz and stress tests

- Graph input fuzz for frontmatter and cyclic dependencies.
- Concurrency tests for bead operations under lock contention.
- Registry import concurrent update/install race.

### 8.5 Performance tests

- Synthetic repo with 1000+ markdown docs:
  - docgraph build and stale computation,
  - `/api/docs/graph` and `/api/docs/stale` latency
  - list endpoint pagination/filters.

### 8.6 Validation gates

- No feature implementation lands without:
  - contract tests,
  - at least one CLI acceptance path,
  - server contract and MCP tests (for new/changed endpoints).

## 10) Risk Register

### R-001: Architecture drift across CLI/server/web
**Likelihood**: High  
**Impact**: High  
**Mitigation**: one shared contract package, schema-driven tests.

### R-002: API key bypass via localhost assumptions
**Likelihood**: Medium  
**Impact**: High  
**Mitigation**: explicit bind-address policy and auth middleware defaults.

### R-003: Frontmatter migration side effects
**Likelihood**: Medium  
**Impact**: Medium  
**Mitigation**: dry-run mode, backups, idempotent write behavior.

### R-004: Registry supply-chain trust assumptions
**Likelihood**: Medium  
**Impact**: High  
**Mitigation**: ADR-003 lockfile + optional additional hash verification + CI audit.

### R-005: File locking contention and partial writes
**Likelihood**: Low  
**Impact**: High  
**Mitigation**: lockfile semantics + retries + tests + fsync on writes.

### R-006: UI complexity outpaces backend contract stability
**Likelihood**: Medium  
**Impact**: Medium  
**Mitigation**: contract-first API versioning and feature flags.

## 11) Observability

### 11.1 Logging

Adopt `slog` with structured fields:

- request_id, method, path, status, duration_ms,
- error code/class, caller source, bytes,
- agent harness, model, token count.

### 11.2 Metrics

- p50/p95/p99 for API latency by route,
- stale checks /graph build duration,
- queue depths for bead operations,
- package verify/install durations,
- MCP request count + error ratio.

### 11.3 Tracing and audit

- Add OTel traces around server routes and registry fetch flows.
- Correlate frontmatter validation failures and install actions.
- Persist immutable event summary for install/verify and server startup parameters.

### 11.4 Operational dashboards

- Readiness:
  - bead store healthy,
  - graph scan healthy,
  - agent harness discovery success,
  - registry cache age.
- Dashboards should show staleness count and critical open P0s.

### 11.5 Alerting thresholds

- 5xx > 1% over 5 minutes.
- Build graph error rate > 2/minute.
- package verify failure rate > 1/minute.
- API auth failure bursts > configurable threshold.

### 11.6 On-call runbook hooks

- Known remediation playbooks for malformed config, missing library path, lockfile mismatch, and harness availability.
