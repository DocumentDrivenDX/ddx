## 2. Planning Stack Consistency

### Findings

**F-001: Most feature specs lack `ddx:` frontmatter** (severity: medium)
- FEAT-001 through FEAT-004 and FEAT-006 have no `ddx:` frontmatter
- Only FEAT-005, FEAT-007, FEAT-008, FEAT-009 have it
- This means they're invisible to the artifact graph they define
- **Resolution:** Add `ddx:` frontmatter to all feature specs

**F-002: Vision and PRD have no `ddx:` frontmatter** (severity: high)
- The two highest-authority documents are not in the artifact graph
- No artifact can `depends_on` them by ID because they have no ID
- **Resolution:** Add frontmatter. Vision gets `id: product-vision`, PRD gets `id: helix.prd`

**F-003: PRD references `ddx:` frontmatter convention but still says `dun:` in some places** (severity: low)
- FEAT-005 examples correctly use `ddx:` 
- FEAT-007 example frontmatter uses `ddx:` correctly
- ADR-001 uses `ddx:` correctly
- But the original FEAT-001 CLI spec predates the rename and doesn't mention frontmatter at all
- **Resolution:** No action needed — the convention is `ddx:`, old specs just predate it

**F-004: No architecture document** (severity: medium)
- ADR-001 and ADR-002 exist (tooling decisions) but there's no architecture doc describing the system's structure, component interactions, or data flow
- Feature specs describe individual capabilities but nothing ties them together architecturally
- **Resolution:** Create an architecture doc covering: CLI structure, server design, doc graph engine, bead storage, agent dispatch, how they compose

**F-005: No test plans** (severity: medium)
- No test plan documents exist for any feature
- Tests exist in code (`cli/internal/bead/*_test.go`, etc.) but they're not governed by test plan artifacts
- Benchmark suite exists (`cli/bench_test.go`) but no performance test plan
- **Resolution:** Defer for now — the code tests are solid. Create test plans when FEAT-002 (server) implementation begins, as that's where integration testing matters most

**F-006: FEAT-001 CLI spec is stale** (severity: high)
- Spec lists user stories for init, browse, personas, sync — these are all implemented
- Spec does NOT mention beads, agent service, or doc graph — all of which are now P0 CLI features
- Spec still references the old `ddx workflow` command in "Out of Scope"
- **Resolution:** Update FEAT-001 to reflect current CLI scope (add bead, agent, doc commands)

**F-007: FEAT-002 Server spec is stale** (severity: medium)
- PRD was updated to include bead endpoints and doc graph endpoints
- FEAT-002 still only describes document browsing and persona resolution
- No mention of bead endpoints, doc graph endpoints, or the web UI integration
- **Resolution:** Update FEAT-002 to include bead, doc graph, and potentially agent log endpoints

**F-008: `internal/artifact` package is orphaned** (severity: low)
- The artifact package (ADR/SD scaffolding) still exists but the CLI commands that used it (`ddx adr`, `ddx sd`) were removed
- Tests still pass but the package is unreachable from any CLI entry point
- **Resolution:** Remove when FEAT-007 (doc graph) is implemented and replaces it, or keep as internal utility if doc graph uses it

## 6. Summary

### What's Working Well
- Bead tracker is complete, bd-compatible, well-tested
- Agent service has core dispatch, quorum, and logging working
- Website is built and deployable
- Feature specs are comprehensive and mostly current
- ADRs for Go and web stack are thorough
- Performance benchmarks established

### Critical Gaps
1. **DDx's own artifacts don't have `ddx:` frontmatter** — eat your own dog food
2. **FEAT-001 CLI spec is stale** — doesn't reflect beads, agents, or doc graph
3. **FEAT-002 Server spec is stale** — missing bead/doc-graph/agent endpoints
4. **No architecture document** — individual features are specified but system design is not
5. **Doc graph (FEAT-007) not implemented** — this is the core "keep documents honest" feature

### Recommended Priority
1. Add `ddx:` frontmatter to all planning artifacts (eat your own dog food)
2. Update FEAT-001 and FEAT-002 to reflect current scope
3. Implement FEAT-007 (doc graph) — it's foundational for everything else
4. Create architecture document
5. Begin FEAT-002 (server) — unlocks FEAT-008 (web UI)
