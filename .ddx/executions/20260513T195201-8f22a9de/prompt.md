    <title>plugin install: symlink local package skill mappings safely</title>
    <description>
PROBLEM&#xA;  Local plugin installs need to make DDx skills live-editable by symlinking package-owned skill directories, but the current local path uses its own discovery rules and can choose the wrong skill source when pointed at the DDx repo root.&#xA;&#xA;ROOT CAUSE&#xA;  cli/cmd/install.go:254-372 implements local install separately from the registry package installer. It loads the manifest but then calls `installLocalSkillSymlinks(absPath, absProject, force)` without passing `pkg.Install.Skills`. cli/cmd/install.go:510-563 chooses `&lt;pluginRoot&gt;/.agents/skills` before `&lt;pluginRoot&gt;/skills`, so `ddx plugin install ddx --local .` can target destination copies in the repo root instead of the intended package source. The plan requires local overlays to honor manifest mappings and either resolve the repo root safely to `library/` or fail before modifying skill outputs.&#xA;&#xA;PROPOSED FIX&#xA;  - cli/cmd/install.go: replace hard-coded local skill discovery with mapping-driven symlink creation from `pkg.Install.Skills`.&#xA;  - Symlink each child skill directory from the mapping source to each mapping target, e.g. `library/skills/ddx -&gt; .agents/skills/ddx` and `.claude/skills/ddx`.&#xA;  - Add repo-root safety: if a local path is not a package root but contains `library/package.yaml` for the requested package, resolve to that package root, or fail clearly before writing outputs.&#xA;  - Preserve local overlay invariants: do not mutate plugin version pins and do not auto-commit.&#xA;&#xA;NON-SCOPE&#xA;  - Remote registry real-file installs; sibling bead owns package mapping copy behavior.&#xA;  - Moving the actual skill content to `library/skills/ddx`.&#xA;  - Changing public plugin command names.&#xA;&#xA;PARENT + DEPS&#xA;  Parent: ddx-6862f2ac. Depends on the registry mapping bead so local and remote installers share mapping semantics.
    </description>
