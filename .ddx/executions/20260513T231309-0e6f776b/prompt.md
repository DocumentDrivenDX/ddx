    <title>work: report live workers only for the requested project</title>
    <description>
PROBLEM&#xA;Operators and agents asking &#34;is the worker still working beads?&#34; or &#34;how is the queue progressing?&#34; can receive the status of a DDx worker from the wrong repository. In the observed incident, the user was asking about the Niflheim queue, but the agent ran a global `ps aux | grep -E &#34;(ddx work|ddx try)&#34;` and reported a DDx repo worker on `ddx-bbdbc903`. This is actively harmful: it makes project A look busy because project B has a worker, and it hides whether project A has stalled.&#xA;&#xA;ROOT CAUSE WITH FILE:LINE&#xA;- `.agents/skills/ddx/reference/status.md:22-31` tells agents how to check queue state with `ddx bead ready`, `ddx bead blocked`, `ddx bead status`, and `ddx bead list`, but it gives no required project-root confirmation or project-scoped live-worker check.&#xA;- `.agents/skills/ddx/reference/status.md:82-99` warns not to mix health/status commands, but does not warn against global process scans or require that worker reports match the requested project root.&#xA;- `.agents/skills/ddx/reference/work.md:143-159` documents `ddx work` and `ddx try`, but does not document a safe way to ask &#34;which live workers belong to this project?&#34;.&#xA;- `cli/cmd/work.go:76` exposes `--project`, and `cli/cmd/work.go:144-165` includes `project_root` in terminal/JSON loop output, but DDx has no corresponding `ddx work status` / `ddx worker status` local CLI that lists live worker processes for the current project and filters out other repositories.&#xA;- `cli/cmd/server_workers.go:13-175` provides server-backed worker listing/logging, but that only helps when the local server worker registry is running; it does not solve ordinary inline `ddx work` shells that operators check from a terminal.&#xA;&#xA;PROPOSED FIX&#xA;Add a project-scoped live-worker status surface and update skill guidance so agents cannot accidentally answer with cross-project workers.&#xA;&#xA;Recommended implementation shape:&#xA;- Add `ddx work status` (or an equivalent `ddx worker status`) that defaults to the current git/project root, accepts `--project`, and lists only live `ddx work` / `ddx try` processes whose process cwd, `--project`, execution worktree, or durable worker record maps to that root.&#xA;- Include `--all-projects` for explicit global inspection. The default must never report workers from other repositories.&#xA;- Output should include project root, pid, age, command, active bead ID when inferable, execution worktree or bundle path when inferable, and a clear `no live workers for &lt;project&gt;` result when none match.&#xA;- Update `.agents/skills/ddx/reference/status.md` and `.agents/skills/ddx/reference/work.md` to require: first resolve the requested project root; then use the project-scoped status command; never use raw global `ps | grep ddx work` as evidence without filtering by project root; always name the project whose worker status is being reported.&#xA;- If a live worker is found for another project while answering a scoped question, mention it only as an explicitly out-of-scope process, not as proof that the requested project is progressing.&#xA;&#xA;NON-SCOPE&#xA;Do not build a full multi-project scheduler. Do not require the DDx server to be running. Do not change bead claiming semantics. Do not kill or restart existing workers. Do not remove the server-backed worker APIs.
