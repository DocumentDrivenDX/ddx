PROBLEM&#xA;`ddx work` does not have a crisp human-facing contract for run modes or stdout. The current docs and code use `--poll-interval` as if it were the mode selector, but every queue worker scans the queue between attempts. Operators need explicit semantics for one-shot work, drain-and-exit work, and long-running watch work, plus output that answers whether the worker exited, why it exited, what queue state remains, and why an idle watcher is sleeping.&#xA;&#xA;ROOT CAUSE WITH FILE:LINE&#xA;- docs/helix/01-frame/features/FEAT-010-task-execution.md:685-702 currently describes the behavior as a `--poll-interval` default, conflating idle sleep tuning with the operator&#39;s chosen work mode.&#xA;- cli/cmd/work.go:37-50 presents `ddx work --poll-interval 30s` as the continuous-worker example rather than naming the mode directly.&#xA;- cli/cmd/work.go:75 exposes `--poll-interval` as the primary stay-alive control, even though `cli/internal/agent/execute_bead_loop.go:649-682` treats any positive interval as idle/watch behavior.&#xA;- cli/cmd/agent_cmd.go:1885-1937 prints terminal summaries without stop reason, already-satisfied breakdown, remaining queue-state context, or blocked-behind-human-review explanation.&#xA;- cli/internal/agent/execute_bead_loop.go:656-688 emits idle progress internally but does not define the human stdout contract for idle or next-ready-bead transitions.&#xA;&#xA;PROPOSED FIX&#xA;Update FEAT-010 and FEAT-006 with an explicit stdout and mode contract:&#xA;- `once`: process at most one execution-ready bead and exit. Public flag remains `--once`.&#xA;- `drain`: process the currently execution-ready queue until a terminal stop condition, then exit. Direct CLI `ddx work` uses this unless `--watch` is supplied.&#xA;- `watch`: process ready work, then remain idle and rescan for newly arriving execution-ready work until signal, fatal config, budget/resource stop, or operator stop. Public flag is `--watch`.&#xA;- `idle interval`: the sleep duration between empty-queue scans while in watch mode. Public flag is `--idle-interval &lt;duration&gt;` with a 30s default in watch mode.&#xA;- `--poll-interval` is removed from the public CLI and generated docs. There is no compatibility alias. New server/API inputs use `mode` + `idle_interval`, not `poll_interval`.&#xA;- Server-managed workers default to watch mode with a 30s idle interval unless the request explicitly asks for `once` or drain mode.&#xA;&#xA;Also define the human stdout contract:&#xA;- Terminal summaries state attempts, closed count, changed/already-satisfied breakdown, failures, stop reason, and remaining queue state.&#xA;- Remaining queue state distinguishes execution-ready, blocked, needs-human, cooldown/deferred, not-eligible, superseded, and epic/closure-candidate buckets where DDx can compute them.&#xA;- Queue summaries explain dependency pressure from human-review blockers: if 30 unique open beads are transitively blocked behind 3 open `needs_human` / needs-investigation blockers, say so and list the blocker IDs/titles with blocked counts.&#xA;- Watch idle output says the worker is still running, gives the idle interval, includes queue-state summary, and uses direct wording such as `idle: no execution-ready beads; sleeping 30s`.&#xA;- Every idle poll prints a compact queue-state line.
