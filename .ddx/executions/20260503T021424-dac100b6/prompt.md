    <title>TD: client-server worker architecture — workers as long-lived API clients; server restart preserves in-flight work</title>
    <description>
Operator-observed gap: when ddx-server restarts, in-flight worker subprocesses lose their connection to the orchestrator. The current model has two execution paths (--local vs server-spawned) that diverge in lifecycle management, configuration plumbing (the dropped-flags class — see ddx-29058e2a), and observability (server can see only what it spawned).&#xA;&#xA;Proposed architectural shift: WORKERS ARE LONG-LIVED PROCESSES THAT PHONE HOME TO THE SERVER VIA API.&#xA;&#xA;DESIGN PROPOSALS (this TD chooses among them)&#xA;&#xA;PROPOSAL A — Server orchestrates, workers are long-lived clients&#xA;- Worker subprocess starts (either via `ddx work` CLI or via server-spawned)&#xA;- Worker registers with server (via /api/workers/register, gets worker-id + heartbeat token)&#xA;- Worker polls /api/workers/&lt;id&gt;/next-bead via long-poll or websocket; receives bead spec from server&#xA;- Worker reports progress via /api/workers/&lt;id&gt;/event {kind, body}&#xA;- Worker reports completion via /api/workers/&lt;id&gt;/result {bead-id, status, evidence}&#xA;- If server restarts: worker&#39;s heartbeat fails → worker enters &#34;disconnected&#34; state, finishes current bead, reconnects on server return&#xA;- If worker dies: server&#39;s heartbeat-timeout reclaims the bead&#xA;&#xA;PROPOSAL B — Server is optional sidecar, worker is autonomous&#xA;- Worker runs independently; no server required&#xA;- If a server is reachable (via .ddx/server.addr), worker emits events to it best-effort&#xA;- Server is a passive observer, not an orchestrator&#xA;- Restart-safe by construction (worker doesn&#39;t depend on server)&#xA;- Loses: server-driven label filters, dynamic priority overrides, centralized cancel&#xA;&#xA;PROPOSAL C — Hybrid: workers are autonomous executors but yield orchestration to server when present&#xA;- Worker checks for .ddx/server.addr at startup&#xA;- If present: registers as managed; receives bead assignments from server&#39;s queue&#xA;- If absent or unreachable: runs autonomously using local bead store&#xA;- Server restart: managed workers detect via heartbeat failure; gracefully complete current bead, then either retry registration OR fall back to autonomous mode (configurable)&#xA;&#xA;UNIFIED API CONTRACT (all proposals)&#xA;&#xA;POST /api/workers/register { project_root, harness, model_pref, label_filter, capabilities[] } → { worker_id, session_token, heartbeat_interval_ms }&#xA;POST /api/workers/&lt;id&gt;/heartbeat { state: idle|claiming|executing|reviewing } → { server_command: continue|pause|drain|terminate }&#xA;GET  /api/workers/&lt;id&gt;/next-bead → { bead, attempt_id, base_rev } | { wait_for_seconds: N }&#xA;POST /api/workers/&lt;id&gt;/event { kind, body, bead_id } → 204&#xA;POST /api/workers/&lt;id&gt;/result { bead_id, attempt_id, status, evidence_dir, commit_sha } → 204&#xA;POST /api/workers/&lt;id&gt;/disconnect → 204 (graceful)&#xA;&#xA;The existing --local mode collapses to: a worker that registers with itself (no server) using an in-process implementation of the API. Same code path, same flags, same lifecycle.&#xA;&#xA;INTEGRATION WITH IN-FLIGHT WORK&#xA;&#xA;This TD intersects:&#xA;- ddx-29058e2a (ExecuteLoopSpec unification): the spec becomes the registration payload — single struct still applies&#xA;- ddx-5cb6e6cd refactor epic: the work package boundary aligns with worker autonomy; C9 (StopCondition) becomes worker-side state&#xA;- ddx-4c51d33e (cross-project leak): workers identify their project at registration; server enforces project boundaries via session token&#xA;- ddx-dc157075 (worker stay-alive): becomes natural — worker stays alive, polls for work, doesn&#39;t exit on empty queue&#xA;&#xA;IMPLEMENTATION OUTLINE (post-TD)&#xA;1. Define API contract + handlers (cli/internal/server/worker_api.go)&#xA;2. Add worker client (cli/internal/agent/work/client.go) implementing the API&#xA;3. Add in-process API implementation for --local mode (cli/internal/agent/work/local_api.go)&#xA;4. Migrate execute_bead_loop / drain_loop to consume the API instead of direct store access&#xA;5. Migrate server worker spawn to register-then-poll instead of inline-execution&#xA;&#xA;NOT IN SCOPE FOR THIS TD&#xA;- Specific code changes (this TD documents the design; implementation is follow-up beads)&#xA;- Multi-server federation (each server owns its workers; cross-server is a separate concern)&#xA;- Auth beyond session tokens (the worker-server channel is local; no remote workers in v1)&#xA;&#xA;OUTPUTS (TD deliverables)&#xA;- ADR document at docs/helix/03-decide/ADR-XXX-worker-client-server-architecture.md&#xA;- Updated FEAT-006 (or new FEAT) describing the worker contract&#xA;- Sequence diagrams for: register, claim, execute, restart-recovery, graceful-shutdown&#xA;- Compatibility table: which existing tests/behaviors break, which migrate cleanly&#xA;- Implementation roadmap: ordered list of beads to file as follow-ups&#xA;&#xA;DEPS&#xA;- Should land BEFORE major refactor children that touch the loop body (C5, C7, C9) — otherwise they&#39;d refactor against the OLD model and need re-doing&#xA;- Should NOT block routine bug fixes or feature beads
    </description>
