Originally observed in ddx v0.6.2-alpha1 against agent v0.9.9 during SD-024 work.&#xA;&#xA;## Symptom&#xA;&#xA;Worker spawned with explicit pin:&#xA;&#xA;    ddx work --harness claude&#xA;&#xA;First attempt on a bead routes correctly to claude (e.g. opus-4.7) and produces a result. When the post-merge reviewer returns REQUEST_CHANGES the bead reopens. The next execute-loop iteration retries the bead but the worker&#39;s --harness override is NOT propagated to that retry. Tier-attempt events show:&#xA;&#xA;    kind:tier-attempt body=tier=standard harness= model= probe=no viable provider, no viable harness found&#xA;    kind:tier-attempt body=tier=smart    harness= model= probe=no viable provider, no viable harness found&#xA;    kind:escalation-summary winning_tier=exhausted&#xA;&#xA;Note harness= and model= empty — the retry constructs a service request with Harness=&#34;&#34;. Smart-routing returns zero candidates at every tier above cheap (cheap suppressed by adaptive min-tier from prior failures). Bead bounces between REQUEST_CHANGES (cycle 1) → execution_failed (cycle 2 with no harness routing) repeatedly.&#xA;&#xA;Concrete instance: ddx-dd7fce7e in /Users/erik/Projects/ddx/.ddx/beads.jsonl shows the pattern at events around 2026-04-25T21:46-21:47Z.&#xA;&#xA;## Principle&#xA;&#xA;**Retries should use the same parameters as the initial work.** A reopened bead&#39;s retry must inherit every explicit pin (harness, model, model-ref, profile, provider) that the worker was spawned with. Otherwise the retry is a different request, not a retry — and the work cannot converge.&#xA;&#xA;This is a general invariant, not specific to harness overrides. Any explicit pin the worker was launched with must propagate to every iteration on every bead the worker handles, including post-reopen retries.&#xA;&#xA;## Diagnosis (from agent-9465287e worker analysis)&#xA;&#xA;The agent service surface (CONTRACT-003) is correct. The agent has no tier concept — tier classification and adaptive min-tier floors are ddx-side abstractions layered on top of the agent&#39;s per-call routing. Loosening the agent&#39;s routing engine to return candidates when the harness is empty would change behavior for every other caller and is not the right lever.&#xA;&#xA;The fix is a few lines of option-propagation in the ddx execute-loop&#39;s reopen-retry path: when a bead reopens after REQUEST_CHANGES (or any review-driven reopen), the loop must copy the worker-level explicit pins into the next iteration&#39;s ExecuteBeadOptions / service.Execute request, the same way it does on the initial claim.&#xA;&#xA;## In-scope files&#xA;&#xA;- cli/internal/agent/execute_bead_loop.go (or wherever the retry path constructs its execute-bead invocation)&#xA;- cli/internal/agent/execute_bead_review.go if review-driven reopens have a separate path&#xA;- Regression test asserting the second service.Execute call carries the same Harness, Model, Profile values as the first when the bead was reopened by a reviewer&#xA;&#xA;## Out of scope&#xA;&#xA;- Agent-side routing engine changes — the surface is correct&#xA;- Adaptive min-tier behavior tuning (separate concern)&#xA;- Tool-loop detection (separate concern, see agent-1bdb9b60 in agent repo)
