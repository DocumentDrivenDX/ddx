    <title>agent: type provider failures and enforce fallback semantics</title>
    <description>
PROBLEM&#xA;DDx/FIZ worker failures can still surface as underspecified execution failures when provider availability, model availability, credentials, endpoint reachability, or harness configuration differs across machines. DDx is the abstraction that should let unattended workers continue through provider variance; a generic provider failure that gives no typed cause, no fallback action, and no durable route-health evidence is a product bug.&#xA;&#xA;ROOT CAUSE WITH FILE:LINE&#xA;cli/internal/agent/service_run.go:287-290 returns pre-dispatch Fizeau Execute errors as a generic `agent: execute: %w`, leaving the execute loop without a typed provider/model/configuration failure contract.&#xA;cli/internal/agent/service_run.go:332-376 copies final service error text into Result.Error/ExitCode, but downstream classification still depends on free-text heuristics rather than a required typed route-health outcome for provider unavailable, model unavailable, auth missing, quota/rate limit, endpoint timeout, or harness misconfiguration.&#xA;cli/cmd/execute_loop_shared.go:516-544 and :588-612 contain the unpinned route-exclusion/escalation path, but that path only works when the failure is already recognized as provider_connectivity/no_viable_provider; untyped provider/configuration failures can stop or preserve the bead without proving fallback was attempted or intentionally impossible.&#xA;&#xA;PROPOSED FIX&#xA;- Define a typed provider failure taxonomy at the DDx/Fizeau boundary: provider_connectivity, provider_auth, provider_rate_limit, provider_quota, provider_model_unavailable, provider_harness_unavailable, provider_config_invalid, no_viable_provider, and unknown_provider_failure.&#xA;- Normalize both pre-dispatch Execute errors and final failed events into that taxonomy before ExecuteBeadReport construction.&#xA;- For unpinned workers, convert retryable provider/model/harness availability failures into route-health evidence and continue fallback/escalation without parking the bead.&#xA;- For explicit operator pins, fail loudly with a typed hard-pin exhausted/operator-action report that names the pin and available remediation, without silently widening pins.&#xA;- Persist durable evidence on the bead/run showing requested constraints, resolved route if any, typed failure, retryability, whether fallback was attempted, and why fallback stopped.&#xA;&#xA;NON-SCOPE&#xA;Do not move concrete route selection into DDx;
