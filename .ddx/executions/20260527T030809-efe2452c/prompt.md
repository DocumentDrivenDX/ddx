    <title>execute-bead workers hold git index.lock for minutes; need lifetime tracking + fail-fast</title>
    <description>
PROBLEM&#xA;ddx work --watch workers running execute-bead can hold the host repo&#39;s .git/index.lock for many minutes during a single bead execution. This serializes every other tracker-affecting operation in that repo (filing beads, dep edits, commits) and produces cascading tracker-lock-timeout failures for operators and other workers.&#xA;&#xA;OBSERVATION (Fizeau, 2026-05-17 21:38–21:44+ EDT)&#xA;- .git/index.lock mtime 21:38; still held at 21:44 = 6+ minutes uninterrupted.&#xA;- Holder: ddx work --watch (PID 90753) running execute-bead for fizeau-4b76e6b8 in worktree .ddx/executions/20260518T013652-2344d0e4.&#xA;- Side-effects observed in the operator session: bead create commands intermittently failed with &#39;tracker lock timeout (max elapsed, lock: /Users/erik/Projects/fizeau/.ddx/.git-tracker.lock, owner pid: 2615737)&#39; and &#39;tracker lock timeout (max elapsed, lock: ..., owner pid: 2809592)&#39;. Both the git index.lock and the .ddx/.git-tracker.lock exhibit the same pattern: held far longer than the operation needs.&#xA;- Five concurrent ddx work --watch daemons (PIDs 87523, 88624, 89333, 90753, 949868) plus a worker-spawned claude subprocess. The lock is held across the entire claude subprocess lifetime, not just the actual git mutation.&#xA;&#xA;WHY THIS IS A BUG&#xA;git&#39;s index.lock and any short-lived mutation lock should be held only for the duration of the mutation itself (single-digit seconds at most). Holding it across the full bead-execution lifetime (which includes a multi-minute LLM subprocess) means:&#xA;1. Any concurrent ddx bead create/update/close from operators or other workers may time out.&#xA;2. Operators cannot do tracker hygiene (annotate findings, file follow-ons) while a long bead runs.&#xA;3. The error surface is misleading — &#39;tracker lock timeout&#39; looks like a tracker bug, but the actual culprit is the index/staging-area lock.&#xA;4. Other ddx work --watch daemons sit idle when they could be making progress on independent beads.&#xA;&#xA;PROPOSED FIX&#xA;A. Scope every git mutation lock to the minimum operation, never across the LLM subprocess invocation. Specifically: do not hold .git/index.lock or .ddx/.git-tracker.lock while ddx is waiting on a claude/codex/gemini subprocess to return.&#xA;B. Instrument lock lifetime — emit a structured metric (lock_name, acquired_at, released_at, duration_ms, holder_pid, operation) for every acquire/release.&#xA;C. Fail-fast threshold — abort with a loud error if any worker holds a lock beyond a configured cap (proposed default: 10s for index.lock, 30s for tracker lock). Hung workers should release and let others through, not block the whole tracker.&#xA;D. CI / load test — a multi-worker scenario test that proves 5 workers can each acquire and release the lock with p99 hold time under the cap.&#xA;&#xA;CROSS-REF&#xA;Filed from Fizeau alignment-review session 2026-05-17 (AR-2026-05-17-repo). Originally surfaced during the no-static-fallback bead chain (parent fizeau-89eca738). No local Fizeau bead — this is a DDx infrastructure bug, filing directly here per AGENTS.md cross-repo triage pattern.
    </description>
