PROBLEM&#xA;The work applies bead-scoped cooldowns to infra-class failures, contradicting documented policy and violating the forward-progress invariant. Observed today: 14 beads all routed to a temporarily-unreachable LM Studio host received 15-min cooldowns in sequence, then the worker went idle with the entire queue gated for 15 minutes despite cloud providers being reachable. A separate worker also hit a 2nd-strike preclaim cooldown after .ddx/beads.jsonl was momentarily staged between two parallel workers&#39; tracker commits — recoverable in milliseconds, but the policy parked the bead for minutes anyway. Both cases produced the failure the operator named: &#34;cooling down every bead in the queue when we are failing to find a provider, how is that going to fix anything?&#34;&#xA;&#xA;ROOT CAUSE — code violates existing spec&#xA;The forward-progress invariant and the right cooldown policy are already documented; the code does not honor them.&#xA;&#xA;- reliability-principles.md P6 (docs/helix/06-iterate/reliability-principles.md:35): &#34;Cooldown fires ONLY when the model genuinely couldn&#39;t make progress (clean no-changes with rationale). Disrupted, preflight-rejected, network-error, claim-race -&gt; no cooldown, return to ready.&#34; Today&#39;s code violates this for `provider_connectivity` and `no_viable_provider`.&#xA;- ADR-024 §Infrastructure And Rate-Limit Fallback (docs/helix/02-design/adr/ADR-024-power-escalation-and-review-routing.md:96): &#34;Infrastructure failures are not capability failures... do not consume escalation budget... apply a retry-after cooldown only when time passing could plausibly fix the same attempt class.&#34; Loose &#34;or&#34; wording lets the current implementation skirt the intent.&#xA;- TD-031 §3.1 (docs/helix/02-design/technical-designs/TD-031-bead-state-machine.md:75): &#34;Continuous forward progress is the default. Before a bead can move to `status=proposed`, the drain loop MUST exhaust every applicable automatic path in priority order.&#34; Says MAY set retry-after for `no_viable_provider`; the code unconditionally sets 15 min regardless of whether alternative routes exist.&#xA;&#xA;Implementation sites that violate the spec:&#xA;- cli/internal/agent/execute_bead_loop.go:2283-2284 — provider_connectivity sets bead cooldown unconditionally after applyProviderConnectivityRouteExclusion already recorded the failed route.
