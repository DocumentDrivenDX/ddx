    <title>ddx work --watch: self-cycle between tasks — re-exec on a newer installed binary so workers never run stale code</title>
    <description>
PROBLEM: `ddx work --watch` runs the binary it was launched with for its ENTIRE lifetime. When newer code is installed (the factory merges fixes + `make install`), the worker keeps executing STALE logic until manually killed+restarted. It only WARNS (&#34;installed ddx is built from X; source HEAD is Y (N commits ahead); run make install&#34;) — no action. Observed live: a 2-worker drain ran ~75 commits behind, missing ALREADY-LANDED concurrency fixes (ddx-33530e68 post-land-outside-lock, ddx-0b659710 atomic-claim-release, ddx-274943fc worktree-leak) and repeatedly colliding on the durable-audit commit — the very fixes were sitting in source but not in the running binary. Operators must manually cycle workers. Operator expectation: &#34;the workers are supposed to know how to cycle automatically between tasks.&#34; (A stopgap supervisor that runs `ddx work --once` + `make install` between cycles is in place; this bead is the durable native fix.)&#xA;&#xA;ROOT CAUSE: the --watch loop loads its binary once at startup; there is no between-task self-refresh / re-exec. The staleness warning (grep &#34;installed ddx is built from&#34; / &#34;source HEAD&#34;) is informational only.&#xA;&#xA;PROPOSED FIX: At a SAFE boundary in the --watch loop — after a bead completes/unclaims and BEFORE claiming the next — detect whether the on-disk installed binary differs from the running process&#39;s build commit (compare the embedded build commit vs the installed binary&#39;s reported commit, or hash/stat the executable resolved via os.Executable/PATH). If newer, re-exec the process in place (syscall.Exec) on the fresh binary, preserving argv/flags (--watch, --min-power, etc.) so the next bead runs on the latest code. NEVER re-exec mid-bead. Gate behind --self-refresh (default ON for --watch). This delivers &#34;workers cycle automatically between tasks.&#34;&#xA;&#xA;NON-SCOPE: triggering `make install`/the build from the worker (separate concern — a supervisor/build pipeline produces the binary; this bead re-execs onto it); the durable-audit collision classifier (ddx-83361480/e8bf0695/2965925e).
    </description>
