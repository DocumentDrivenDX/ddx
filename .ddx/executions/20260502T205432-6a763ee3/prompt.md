    <title>ddx work: server path drops 5 CLI flags (opaque_passthrough, max-cost, request-timeout, min-power, max-power)</title>
    <description>
ddx work / ddx agent execute-loop has two execution paths: --local (inline) and default (submit to running server). The server path silently drops five CLI flags between client → server request → ExecuteLoopWorkerSpec → spawned worker subprocess. Most user-visible symptom: `ddx work --harness claude` fails when going through the server but works under --local, because the server validates the harness against its provider list instead of treating it as an opaque passthrough constraint.&#xA;&#xA;USER-OBSERVED SYMPTOM&#xA;- `ddx work --harness claude --local` works.&#xA;- `ddx work --harness claude` (server path) fails.&#xA;- Server path is supposed to be a transparent passthrough — same flags, same behavior.&#xA;&#xA;ROOT CAUSE: chain of omissions&#xA;&#xA;1. CLIENT — cli/cmd/agent_cmd.go:1608-1611 extracts maxCostUSD, requestTimeout, minPower, maxPower from the cobra command but cli/cmd/agent_cmd.go:1627 (call to executeLoopWithServer) and cli/cmd/agent_cmd.go:1941 (the function signature) do NOT include them. The four flags are dropped client-side before any HTTP call.&#xA;&#xA;2. CLIENT → SERVER — opaque_passthrough IS sent in the worker spec at cli/cmd/agent_cmd.go:1948-1950, BUT:&#xA;&#xA;3. SERVER REQUEST STRUCT — cli/internal/server/server.go:2246-2260 (handleStartExecuteLoopWorker) request struct is MISSING fields for OpaquePassthrough, MaxCostUSD, RequestTimeout, MinPower, MaxPower. Server JSON-decode silently drops the unknown fields.&#xA;&#xA;4. SERVER → SPEC — cli/internal/server/server.go:2289-2303 (call to StartExecuteLoop) does not set OpaquePassthrough on the spec (since it never decoded it). MaxCostUSD/RequestTimeout/MinPower/MaxPower aren&#39;t even spec fields yet.&#xA;&#xA;5. WORKER SPAWN — cli/internal/server/workers.go:656 hardcodes escalation.DefaultMaxCostUSD instead of spec.MaxCostUSD. RequestTimeout override never applied.&#xA;&#xA;6. VALIDATION DIVERGENCE — at cli/cmd/agent_cmd.go:1647-1652, --local correctly skips harness validation when treatPassthroughAsOpaque=true (the ddx work case). On the server path, the same skip never happens because OpaquePassthrough was lost in step 3 — so harness &#39;claude&#39; goes through ValidateForExecuteLoopViaService which rejects it. **This is the user&#39;s observed bug.**&#xA;&#xA;WHY THIS IS A REAL BUG (not &#34;just configure the server&#34;)&#xA;The server should be a transparent passthrough — every CLI flag accepted by ddx work should produce identical behavior whether routed --local or via the server. Silently dropping flags violates the contract and produces confusing failures (the local path works, the server path doesn&#39;t, with no error explaining why).&#xA;&#xA;FIX SHAPE&#xA;- Add fields to server request struct (server.go:2246-2260): OpaquePassthrough bool, MaxCostUSD float64, RequestTimeout string, MinPower int, MaxPower int&#xA;- Add parameters to executeLoopWithServer signature (agent_cmd.go:1941); pass maxCostUSD/requestTimeout/minPower/maxPower from caller (agent_cmd.go:1627)&#xA;- Add fields to ExecuteLoopWorkerSpec where missing (workers.go:25-49) — MaxCostUSD/RequestTimeout/MinPower/MaxPower&#xA;- Populate spec from request in handler (server.go:2289-2303)&#xA;- Use spec values in runWorker (workers.go:656 cost cap; ~594-601 request-timeout override)&#xA;&#xA;NOT IN SCOPE&#xA;- Reworking the client→server transport (HTTP/JSON stays as-is; just add fields).&#xA;- Changing the validation logic itself — only ensuring opaque_passthrough actually gates validation as intended.
