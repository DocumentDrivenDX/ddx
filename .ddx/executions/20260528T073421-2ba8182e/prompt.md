    <title>readiness: discount stale external blockers cleared by newer notes</title>
    <description>
PROBLEM&#xA;The readiness hook can over-weight stale external-blocker history and under-weight explicit operator notes that say the blocker was cleared. This produces misleading `intake.warn` text and can steer an implementation attempt toward obsolete infrastructure conclusions even when the bead is intentionally unblocked for forward progress.&#xA;&#xA;OBSERVED EVIDENCE&#xA;Niflheim bead `niflheim-439a49ae` has notes saying: `Unblocked 2026-05-18: the prior Redpanda/AIO blocker was stale after commit 90cd8348 bounded Redpanda test storage with an owned tmpfs mount... The container became ready and was cleaned up.` During `ddx work --once` on 2026-05-21, readiness still emitted `bead readiness blocked: redpanda environment can support 32-partition stress test... prior_attempts show a hard external blocker: Redpanda fails to start due to AIO slot exhaustion on WSL2`. The attempt continued because readiness was warn-only, but the warning contradicted current bead notes and polluted the agent context.&#xA;&#xA;ROOT CAUSE WITH FILE:LINE&#xA;`cli/internal/agent/triage_hook.go:60` includes bead notes in the triage/readiness payload model, and `cli/internal/agent/types.go:298` carries notes in the execute-bead payload, but the readiness prompt/contract does not require stale prior blockers to be discounted when a later note explicitly clears them. The current tests in `cli/internal/agent/execute_bead_intake_test.go` cover warn-only continuation and intake event recording, but not precedence between later unblocked notes and older `no_changes_blocked`/external-blocker events.&#xA;&#xA;PROPOSED FIX&#xA;Update the readiness prompt/contract and pre-claim intake tests so later explicit operator notes or reopen/unblocked notes supersede older external-blocker events unless current evidence revalidates the blocker. Readiness may still mention stale blocker history as historical context, but it must not phrase the bead as currently blocked or require operator action when a newer note says the blocker was cleared and no fresh recheck failed.&#xA;&#xA;NON-SCOPE&#xA;- Changing lifecycle status semantics for actual `status=blocked` beads.&#xA;- Suppressing all prior-attempt history.&#xA;- Removing warn-only readiness behavior.&#xA;- Niflheim benchmark implementation.&#xA;&#xA;IN-SCOPE FILES&#xA;- `cli/internal/agent/execute_bead_intake_test.go`&#xA;- `cli/internal/agent/bead_lifecycle_skill_contract_test.go`&#xA;- DDx bead-lifecycle skill text under `.agents/skills/ddx/bead-lifecycle/SKILL.md` and `.claude/skills/ddx/bead-lifecycle/SKILL.md` if the prompt contract lives there.&#xA;- Any pre-claim intake prompt assembly code that formats notes/events for readiness.
    </description>
