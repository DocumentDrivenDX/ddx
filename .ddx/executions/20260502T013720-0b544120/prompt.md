    <title>ddx work should auto-route execution without project config</title>
    <description>
Operators should not need to author any project-level routing config (.ddx/config.yaml, --harness, --profile, --model, --provider) for `ddx work` to drain a project&#39;s bead queue. The platform should auto-select harness, model, and provider based on bead complexity, retry/fall back on failures, and decompose work that is too coarse for one dispatch.&#xA;&#xA;## Observed behavior (HELIX project, 2026-05-01)&#xA;&#xA;Running `ddx work --once --local` from `/Users/erik/Projects/helix`:&#xA;&#xA;```&#xA;warning: triage complexity gate is disabled; coarse beads may waste dispatch attempts&#xA;▶ helix-13cfe23f: Align HELIX execution contract to shipped DDx execute-loop / execute-bead capabilities&#xA;✓ helix-13cfe23f → error: routing under-specified: pass at least one of Harness/Profile/ModelRef/Model/Provider, or auto-selection inputs&#xA;project: /Users/erik/Projects/helix&#xA;completed: 1  |  successes: 0  |  failures: 1&#xA;```&#xA;&#xA;Workarounds tried, all failed or were rejected by users as the wrong layer to address this:&#xA;&#xA;1. `ddx work --once --local --profile cheap` — same routing-under-specified error.&#xA;2. `ddx work --once --local --harness openrouter` — `harness &#34;openrouter&#34;: no configured provider matches type &#34;openrouter&#34; (available: [])`, despite `~/.config/agent/config.yaml` having multiple openrouter-typed providers.&#xA;3. `--min-tier cheap` — unknown flag.&#xA;&#xA;By contrast, the same `ddx work` invocation succeeds for the DDx project itself — workers in `ddx server workers list` show recent successful runs against ddx-* beads. So the per-project config gap is the difference, not anything global.&#xA;&#xA;## Expected behavior&#xA;&#xA;`ddx work` (and `ddx agent execute-loop` directly) should:&#xA;&#xA;1. Auto-select routing for each bead with no flags and no project config required, given that the global agent config (`~/.config/agent/config.yaml`) has providers configured.&#xA;2. Make routing decisions from bead complexity: doc-only or single-file changes → cheap tier; multi-file refactors → standard; cross-cutting work → smart. The &#39;triage complexity gate&#39; warning suggests this exists but is disabled by default.&#xA;3. Retry on transient failure within the same tier; escalate to a higher tier on substantive failure (this is documented behavior — `winning_tier=exhausted` events show the escalation chain works once dispatched, but auto-routing has to dispatch first).&#xA;4. When a bead is too coarse for any tier to execute reliably, either decompose it into child beads or surface a &#39;needs operator triage&#39; status with a clear reason — never silently wedge with `routing under-specified`.&#xA;5. Default to using the global agent config&#39;s `default_provider` when no project override exists, instead of treating the absence of project config as a hard error.&#xA;&#xA;## Why this matters&#xA;&#xA;- **Adoption friction**: Every new project that adopts DDx hits this wall. Authoring `.ddx/config.yaml` requires understanding DDx&#39;s routing internals before an operator can run `ddx work` once.&#xA;- **Methodology coupling**: HELIX&#39;s value proposition includes &#39;supervised autopilot&#39; — the autopilot is supposed to drain ready beads. When `ddx work` requires per-project routing config, HELIX projects have to ship operational setup that distracts from the methodology.&#xA;- **Operator surprise**: The DDx-vs-HELIX boundary contract (HELIX/CONTRACT-001) explicitly says HELIX should not own routing config. But projects can&#39;t run `ddx work` without it. That&#39;s a contract violation manifesting as setup burden.&#xA;- **Triage gate disabled by default**: The warning &#39;triage complexity gate is disabled; coarse beads may waste dispatch attempts&#39; is emitted on every run but the operator has no obvious way to enable it. The default should be enabled with a documented opt-out, not the inverse.&#xA;&#xA;## Suggested approach (not prescriptive)&#xA;&#xA;1. Audit the auto-selection input requirements: what inputs are needed to pick a tier when no project config is present? Likely the bead&#39;s labels, `kind`, file-touch estimate, and prior-attempt history. All available without operator input.&#xA;2. Make &#39;use global agent config defaults&#39; the fallback when project routing is unspecified — currently the code requires explicit project routing OR explicit flags.&#xA;3. Enable triage complexity gate by default; document the opt-out flag.&#xA;4. Add an integration test: `ddx work --once --local` succeeds against a fresh project with no `.ddx/config.yaml` and a single bead in `.ddx/beads.jsonl`, given a minimal `~/.config/agent/config.yaml` with one provider.&#xA;&#xA;## Out of scope&#xA;&#xA;- Changing the global agent config schema. The fix is to reduce dependency on per-project config, not redesign the global one.&#xA;- Per-bead model overrides for advanced users — those should remain available via flags but should not be required for the common case.&#xA;&#xA;## Reference&#xA;&#xA;- HELIX alignment review where this surfaced: `docs/helix/06-iterate/alignment-reviews/AR-2026-05-01-public-examples.md` (in the easel/helix repo). The audit&#39;s 12 follow-up beads were drained via direct sub-agent dispatch instead of `ddx work` because of this gap.
    </description>
    <acceptance>
1. `ddx work --once --local` succeeds (exits 0 with at least one successful bead close OR a clean &#39;no ready work&#39; exit) on a project that has only `.ddx/beads.jsonl` and no `.ddx/config.yaml`, when the global agent config has at least one provider configured.&#xA;&#xA;2. `ddx work` does not emit `routing under-specified` for any bead unless the global agent config is itself empty — that case must produce a clear error pointing at the global config, not a per-project routing flag list.&#xA;&#xA;3. Triage complexity gate is enabled by default. The warning &#39;triage complexity gate is disabled; coarse beads may waste dispatch attempts&#39; must not appear in the default invocation. Opt-out flag exists and is documented.&#xA;&#xA;4. Auto-tier selection picks a tier from bead metadata alone (labels, kind, estimated scope) for beads that have those fields, with the global default_provider for routing when no per-project config is present.&#xA;&#xA;5. Coarse-bead handling: when no tier can confidently execute a bead (e.g. it spans more than N files or N hours of estimated work), DDx either decomposes the bead (records this as decomposition events on the bead) or emits a `needs_triage` status (not `execution_failed`) with a clear reason.&#xA;&#xA;6. Failure recovery: transient failures (network, rate limit, timeout) retry within the selected tier; substantive failures (probe_ok+exec_fail or harness errors) escalate to the next tier per existing escalation logic.&#xA;&#xA;7. Integration test in the DDx test suite: `tests/integration/zero-config-work_test.go` (or equivalent) creates a temp project with one bead and a minimal global config, runs `ddx work --once --local`, asserts the bead closes successfully with no project config files created.&#xA;&#xA;8. Documentation update: `docs/getting-started.md` (or equivalent) shows the zero-config flow as the default, with project config presented as an advanced override rather than a prerequisite.
    </acceptance>
