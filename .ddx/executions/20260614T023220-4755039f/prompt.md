    <title>agent: supervise background monitor shells during execute-bead</title>
    <description>
PROBLEM&#xA;During dogfood of ddx-3b721804 on 2026-06-13, a Claude execute-bead attempt repeatedly spawned shell monitor commands such as `until ! kill -0 $(pgrep -f &#34;lefthook run pre-commit&#34;) ...`. After the real `lefthook`/`go test` process exited, the watcher shells remained alive because `pgrep -f` matched the watcher command text itself. The worker made no progress until the operator killed only those watcher PIDs. DDx must not let provider/tool monitor children create an unbounded hidden stall.&#xA;&#xA;ROOT CAUSE WITH FILE:LINE&#xA;cli/internal/agent/agent_runner_service.go:337-366 arms idle and tool-call timers around service events, but it does not track or bound shell children that remain alive after the useful tool process exits.&#xA;cli/internal/agent/execute_bead.go:1992-1996 tells agents to run tests and pre-commit, but the execute-bead contract does not prohibit self-matching background watchers or require bounded monitor commands.&#xA;cli/internal/agent/execute_bead.go:2010 tells the agent to stop after the commit, but DDx has no enforcement that stale background monitor shells are terminated and reported before the provider resumes.&#xA;&#xA;PROPOSED FIX&#xA;Add attempt-scoped supervision for background shell/monitor children launched under the execution worktree. Detect stale monitor commands whose watched process has exited or whose command pattern matches themselves, terminate them without killing the main provider process, and record durable evidence. Expose the stale command count/oldest age in worker status or attempt evidence so operators do not have to inspect `ps`.&#xA;&#xA;NON-SCOPE&#xA;Do not ban all background commands. Do not assume only Claude can produce this pattern. Do not rely on real provider CLIs in tests.
    </description>
