    <title>Persistent drain-queue worker indicator in nav + N-worker count control on dashboard</title>
    <description>
## Observed / Motivation&#xA;&#xA;Today the queue-drain worker is invisible unless the operator is on the project home page or the Workers tab. An operator on any other page (Beads, Documents, Efficacy) has no idea whether a drain is active, how many ready beads are left, or whether they should kick off more capacity. The per-project home page&#39;s &#34;Drain queue&#34; button is one-shot and offers no scaling control — you start one worker and that&#39;s it.&#xA;&#xA;User feedback: the drain should be a **top-level always-visible affordance** with:&#xA;1. A persistent spinner/badge in the global nav showing drain state + an at-a-glance count (ready beads, running workers).&#xA;2. Clicking the badge goes to a workers overview where the operator can see what each worker is doing.&#xA;3. That overview needs a &#34;number of workers&#34; control — `Add worker` / `Remove worker` to scale drain capacity up or down, not just start-one-at-a-time.&#xA;&#xA;## Why this is a new surface, not just a tweak to existing Workers page&#xA;&#xA;- **Global spinner** is a nav-shell concern, not a page concern. It needs to be mounted in `NavShell.svelte` so every route sees it.&#xA;- **Worker-count control** is a different interaction model from the per-worker Start/Stop flow proposed in `ddx-69789664`. That flow is &#34;pick a harness + effort + filter, start one worker with those specs.&#34; This flow is &#34;I want N general-purpose drain workers; add/remove one at a time.&#34; Both should coexist.&#xA;- A **worker-count affordance** implies a default worker spec (default harness, default profile). That&#39;s fine — it mirrors `ddx work` with no flags — but it has to be explicit so the operator knows what they&#39;re spawning.&#xA;&#xA;## Scope&#xA;&#xA;### Part 1 — Global drain indicator in `NavShell`&#xA;&#xA;- New element in the top nav bar, visible on every route (fits next to the theme toggle).&#xA;- Two states:&#xA;  - **Idle** (zero running drain workers): subtle dot + &#34;Queue: X ready&#34; (where X is project-scoped ready-bead count). Clicking navigates to the workers view.&#xA;  - **Active** (≥1 running drain worker): animated spinner + &#34;N workers · X ready&#34;. Clicking navigates to the workers view.&#xA;- Data: new lightweight query `queueAndWorkersSummary(projectId) { readyBeads, runningWorkers, totalWorkers }`, subscribes to worker-progress events to stay fresh.&#xA;- Scope: tied to the currently-selected project in the `ProjectPicker`. If no project is selected, the indicator is hidden.&#xA;- Perf: the query must be cheap — must not trigger the N·M scan already tracked in `ddx-9ce6842a`. Depends on those fixes for the ready-bead count.&#xA;&#xA;### Part 2 — Workers overview with count control&#xA;&#xA;- Repurpose the existing `/workers` route (or add a separate `/workers/overview` — design call) to include a header section with:&#xA;  - Large &#34;Drain workers: N&#34; number.&#xA;  - `+` button (&#34;Add worker&#34;) — dispatches a new default-spec drain worker via the existing `workerDispatch(kind: &#34;execute-loop&#34;)` mutation.&#xA;  - `−` button (&#34;Remove worker&#34;) — stops the oldest-running drain worker. Confirm-gated.&#xA;  - Small help text: &#34;Adds a general-purpose queue-drain worker. Use the per-harness picker below for custom specs.&#34;&#xA;- The existing per-worker table and per-row Start/Stop (from `ddx-69789664`) remain for fine control.&#xA;- &#34;Default spec&#34; is: harness unset (route via profile), profile `default`, no effort override. Operator can change the project default in config.&#xA;&#xA;### Part 3 — Defaults + config&#xA;&#xA;- `.ddx/config.yaml: workers.default_spec` — object with optional harness, profile, effort, min_tier, max_tier. Used by Part 2&#39;s `+ Add worker`. Unset = use `ddx work` defaults.&#xA;- `.ddx/config.yaml: workers.max_count` — optional safety rail, default unset. When set, `+ Add worker` refuses past the cap and tooltips &#34;at workers.max_count limit&#34;.&#xA;&#xA;### Part 4 — Empty/error states&#xA;&#xA;- Indicator on a project with no ready beads and zero workers: &#34;Queue: 0 ready — nothing to drain&#34;. Clicking still navigates (so operator can spawn one anyway).&#xA;- Indicator when the per-project workers query is broken (pre-`ddx-05b4cc9d`): falls back to the global count; logs a one-time console warning.&#xA;- Add-worker failure: surfaces inline on the overview page (error toast + detail), doesn&#39;t silently drop.&#xA;&#xA;## Out of scope&#xA;&#xA;- Per-harness-specific queue views (e.g., &#34;drain with codex&#34; as a distinct indicator).&#xA;- Bead prioritization rules for multi-worker draining — beads already have a priority field, and `execute-loop` already claims in ready-order. No change.&#xA;- Concurrency safety review of multi-worker draining. It already works (beads are claimed atomically), but if `ddx-05b4cc9d` surfaces issues there, file a follow-up.
    </description>
    <acceptance>
**User story:** As an operator on any page in the project, I always see whether a drain worker is running and how much work is queued. One click takes me to the worker overview where I can scale capacity up or down with a single button, without picking a harness or profile every time.&#xA;&#xA;**Acceptance criteria:**&#xA;&#xA;1. **Nav indicator visible everywhere.**&#xA;   - A component in `NavShell.svelte` renders the drain indicator on every route inside a selected project.&#xA;   - Idle state: dot + &#34;Queue: X ready&#34;.&#xA;   - Active state: spinner + &#34;N workers · X ready&#34;.&#xA;   - Clicking navigates to the workers overview.&#xA;   - Playwright: navigate to beads, documents, sessions, efficacy — assert indicator is present on each with the expected text.&#xA;&#xA;2. **Live updates.**&#xA;   - The indicator updates within 2s when a worker starts, finishes, or changes state. Backed by the existing worker-progress subscription.&#xA;   - When ready-bead count changes (a bead is created/closed), the indicator updates within 5s. Either subscription-driven or short poll.&#xA;&#xA;3. **Summary query.**&#xA;   - New `queueAndWorkersSummary(projectId) { readyBeads runningWorkers totalWorkers }`. Returns zeros on unknown or empty projects — no error.&#xA;   - Resolver is cheap: reads the ready count from the same path bead-status queries use, not by iterating every bead.&#xA;&#xA;4. **Overview page — count control.**&#xA;   - Header shows &#34;Drain workers: N&#34; prominently.&#xA;   - `+ Add worker` button dispatches a default-spec worker (`workerDispatch(kind: &#34;execute-loop&#34;, args: null)`). On success the row appears in the table within 2s and the indicator updates.&#xA;   - `− Remove worker` button stops the oldest-running drain worker. Confirm-gated (&#34;Stop worker-XYZ?&#34;). After confirm, the worker transitions to `stopped` within 2s.&#xA;   - Help text below the buttons: &#34;Adds a general-purpose drain worker. Use the per-harness picker below for custom specs.&#34;&#xA;&#xA;5. **Default-spec + safety rail.**&#xA;   - `.ddx/config.yaml: workers.default_spec` is respected by `+ Add worker`. Integration test asserts a configured `profile: cheap` is propagated.&#xA;   - `.ddx/config.yaml: workers.max_count`: when set, `+ Add worker` refuses past the cap; button disabled with tooltip &#34;at workers.max_count limit&#34;. Not set → no cap. Integration test covers both.&#xA;&#xA;6. **Playwright — flow.**&#xA;   - Seeds a project with 3 ready beads, zero workers.&#xA;   - Asserts indicator shows &#34;Queue: 3 ready&#34;, dot state.&#xA;   - Clicks indicator → navigates to overview.&#xA;   - Clicks `+ Add worker` → asserts new row and indicator transitions to spinner + &#34;1 worker · 3 ready&#34;.&#xA;   - Clicks `+` again → asserts &#34;2 workers · 3 ready&#34;.&#xA;   - Clicks `− Remove worker` → asserts one worker stops and indicator shows &#34;1 worker · ...&#34; within 2s.&#xA;   - Navigates to /beads → asserts indicator persists and stays live.&#xA;&#xA;7. **No regressions.**&#xA;   - Existing Drain button on project home continues to work (dispatches the same kind). Can stay or be removed depending on design note; recommend keeping it as the initial-kickoff affordance when zero workers run.&#xA;   - Existing Workers table rendering + live-phase subscription is unchanged.&#xA;   - Existing per-worker Start/Stop from `ddx-69789664` coexists unchanged.&#xA;&#xA;8. **Cross-references.**&#xA;   - Hard dependency: `ddx-05b4cc9d` (fixes `workersByProject` filter — without it, the running-workers count is always 0 on a per-project query).&#xA;   - Adjacent: `ddx-69789664` (per-worker lifecycle + Sessions IA). These beads are complementary; implementation should not re-introduce either&#39;s functionality, just use it.&#xA;   - Adjacent: `ddx-9ce6842a` (perf harness). The `queueAndWorkersSummary` resolver gets a perf target in that harness: p95 ≤ 30ms in-process on a 5k-bead fixture. No full scan allowed.
    </acceptance>
