    <title>execute-bead: prevent checkpoints from absorbing implementation changes</title>
    <description>
PROBLEM&#xA;Live `ddx work` attempts are landing substantial implementation changes inside `chore: checkpoint pre-execute-bead ...` commits instead of the required conventional substantive commit tagged with `[ddx-&lt;id&gt;]`. This weakens the execute-bead audit trail: cost, latency, retry, and review tooling can no longer reliably treat the checkpoint as tracker/caller-dirt bookkeeping and the tagged commit as the bead implementation.&#xA;&#xA;ROOT CAUSE WITH FILE:LINE&#xA;- `cli/internal/agent/execute_bead.go:542-599` defines `checkpointPreDispatchDirt`, which stages `git add -A -- .` through a temporary index and creates a `chore: checkpoint pre-execute-bead &lt;attempt&gt;` commit.&#xA;- `cli/internal/agent/execute_bead.go:1050-1064` runs `commitTrackerLocked` and then `checkpointPreDispatchDirt` before resolving the base revision and creating the isolated execute-bead worktree.&#xA;- `cli/internal/agent/execute_bead_concurrent_predispatch_test.go:17-29` and `:124-138` assert the lock-scope and HEAD-race behavior, but not the content contract for checkpoint commits.&#xA;- Observed evidence: `df5b591ec chore: checkpoint pre-execute-bead 20260513T175830-e116edd8` contains 305 lines of readiness-timeout implementation across seven code/test files, and `c53f2b0fe chore: checkpoint pre-execute-bead 20260513T183205-80b61705` contains 170 lines of `readiness_checks` implementation across three files. The later tagged commit for `ddx-90b84d73` contained only follow-on CLI/docs plumbing, so the main implementation was attributed to the checkpoint.&#xA;&#xA;PROPOSED FIX&#xA;Add a checkpoint content guard so pre-dispatch checkpoint commits cannot silently absorb ordinary implementation files. The guard should either: (a) restrict checkpoints to known durable tracker/evidence/caller-dirt paths and reject code/test paths with an actionable execution failure, or (b) require dirty parent checkout implementation files to be committed under the active bead&#39;s `[ddx-&lt;id&gt;]` commit before checkpointing. Preserve the existing serialized lock behavior that prevents concurrent HEAD CAS failures.&#xA;&#xA;NON-SCOPE&#xA;Do not remove the pre-dispatch checkpoint mechanism entirely. Do not weaken the existing concurrent worker lock tests. Do not rewrite existing history to repair already-landed checkpoint commits.
