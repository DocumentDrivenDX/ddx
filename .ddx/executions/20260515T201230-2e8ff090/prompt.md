    <description>
PROBLEM&#xA;When an operator wants to retry a bead currently in cooldown (work-retry-after set in the future), the only way today is to manually run ddx bead update &lt;id&gt; --unset work-retry-after before each ddx try / ddx work invocation. With 14 beads in cooldown today the operator ran 14 such commands in sequence (and even then a single ddx bead clear-cooldown --all doesn&#39;t exist yet — that&#39;s ddx-63688d7b). There is no CLI surface for &#34;I know this bead is cooled; run it anyway because I just resolved the underlying issue.&#34;&#xA;&#xA;ROOT CAUSE&#xA;cli/cmd/work.go and cli/cmd/try.go both honor work-retry-after via the standard ready-queue filter. There is no override flag. The escalation ladder and route-exclusion machinery is correct (and ddx-63688d7b will narrow when cooldowns even apply), but operators still need a per-invocation escape hatch for the cases where cooldown was set by a now-resolved condition.&#xA;&#xA;PROPOSED FIX&#xA;- Add --force-claim to ddx try &lt;id&gt;: ignore work-retry-after for the specified bead during this attempt. Emit a kind=force-claim event with actor=&lt;assignee&gt; and the timestamp the override was used. The cooldown field is NOT cleared; if the attempt fails again, the original cooldown remains in effect for queued workers.&#xA;- Add --ignore-cooldown to ddx work: drain mode, ignore work-retry-after across the queue for this worker session. Same event semantics.&#xA;- Both flags emit a structured event so the audit trail shows &#34;operator forced through cooldown at time T&#34;.&#xA;- Both flags require an explicit --reason &#34;&lt;text&gt;&#34; string (operator must explain why). The reason is captured in the event body.&#xA;&#xA;NON-SCOPE&#xA;- Do not change the cooldown-setting policy (ddx-63688d7b owns that).&#xA;- Do not auto-clear cooldowns after a successful forced attempt (the bead&#39;s success implicitly invalidates any stale cooldown via its own state, but we don&#39;t mutate the field).&#xA;- Do not expose this flag in passthrough constraints; it&#39;s an operator override only.&#xA;&#xA;DEPS&#xA;ddx-63688d7b (the cooldown-policy rework).
