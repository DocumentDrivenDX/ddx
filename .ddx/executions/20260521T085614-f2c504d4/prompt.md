PROBLEM: There is no way to enqueue review-finding or alignment-review work as first-class tracker beads. Today review/alignment runs as post-cycle hooks invoked directly by the HELIX driver, bypassing the ready queue, scheduler, and standard evidence bundle. Without a typed bead kind and an injection entry point, execute-loop cannot treat them as ordinary work units.&#xA;&#xA;ROOT CAUSE: internal/tracker/kinds.go:1 — bead-kind enum only covers human-authored kinds (feature, story, task); no `review-finding` or `alignment-review` kind exists. internal/tracker/queue.go:1 — Queue exposes Pop/Push for the loop but no programmatic Inject(parent, kind, payload) used by HELIX itself. internal/helix/exec/loop.go:1 — execute-loop dispatch table has no handler that produces the canonical evidence bundle for these kinds.&#xA;&#xA;PROPOSED FIX: (a) Add `BeadKindReviewFinding` and `BeadKindAlignmentReview` to the kind enum with validation rules (parent must be the bead under review/align; payload schema documented). (b) Add `tracker.Inject(ctx, parent, kind, payload) (BeadID, error)` that creates the bead in state `ready`, idempotent on (parent, kind, payload-hash). (c) Add execute-loop dispatchers `runReviewFinding` and `runAlignmentReview` that invoke the existing review/align engines but write results through the standard evidence-bundle writer so downstream artifacts (logs, diffs, attestations) match other bead executions.&#xA;&#xA;NON-SCOPE: HELIX driver wiring (covered by children 2 and 3).
