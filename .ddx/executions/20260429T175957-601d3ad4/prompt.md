    <title>agent workers: stale &#39;running&#39; entries can&#39;t be cleaned without server restart</title>
    <description>
When a local execute-bead/execute-loop process exits abnormally (killed, OOM, parent shell terminated, container crash) the worker entry can remain in the in-memory registry forever. `ddx agent workers` lists it as state=running with elapsed_ms continuously advancing, even though no PID exists.&#xA;&#xA;Reproducing today (axon repo): worker local-20260429T030828-075a28c5 (bead hx-5f4d4ea1, harness=agent, model=qwen3.6) started at 03:08 UTC, has logged elapsed=10+ hours, and `ps -ef | grep execute-bead` shows nothing. The git worktree at /tmp/ddx-exec-wt/.execute-bead-wt-hx-5f4d4ea1-* has only the checkpoint commit (no actual work commits), confirming the worker died early.&#xA;&#xA;Available paths to clean up are all unsatisfying:&#xA;1. `ddx agent workers stop --bead hx-5f4d4ea1` returns &#39;no matching workers&#39; (because the matcher only finds live workers)&#xA;2. `ddx agent workers stop &lt;worker-id&gt;` returns server error 400 &#39;worker not running&#39; (same contradiction — the registry says running, the stopper says not running)&#xA;3. `systemctl --user restart ddx-server` works but kills every other active worker on the box, including ones for other repos that are doing real work&#xA;4. Manual edit of server-state.json risks corruption and isn&#39;t documented&#xA;&#xA;The bead itself wasn&#39;t blocked (status=open, owner=null) so the registry noise is mostly cosmetic — it pollutes `ddx agent workers` output and confuses operators trying to triage the queue. But over weeks of use these accumulate (saw 14+ stale entries from prior sessions on this same box, with ages 134h–268h).
    </description>
    <acceptance>
AC1. `ddx agent workers prune` (or equivalent) exists and reaps registry entries whose recorded PID is no longer alive (or whose started_at is older than a configurable threshold). Output lists each reaped worker with id, bead_id, harness, age.&#xA;&#xA;AC2. `ddx agent workers stop --bead &lt;id&gt;` succeeds against a stale registry entry too — it should release the registry slot even when the underlying process has already exited. Current &#39;no matching workers&#39; / &#39;worker not running&#39; contradiction must be resolved.&#xA;&#xA;AC3. The local-worker registry persists or is reconstructed in a way that survives ddx-server restarts cleanly: either persisted with PID liveness check on restart, or rebuilt from execute-bead/execute-loop on-disk evidence (.ddx/executions/&lt;id&gt;/result.json).&#xA;&#xA;AC4. `ddx agent workers --json` includes a &#39;liveness&#39; or &#39;pid_alive&#39; field per entry so operators can see at a glance which entries are live vs stale, even before pruning.
    </acceptance>
