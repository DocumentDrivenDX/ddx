    <title>cli: translate SIGINT/SIGTERM into graceful ddx cancellation</title>
    <description>
PROBLEM&#xA;Ctrl-C / SIGTERM should trigger DDx cooperative cancellation for `ddx work` and `ddx try`, but the CLI root currently executes without a signal-backed context. Operators see abrupt termination rather than an explicit shutdown message and cleanup path.&#xA;&#xA;ROOT CAUSE WITH FILE:LINE&#xA;- cli/main.go:31 calls `cmd.Execute(workingDir)` and relies on the default process signal behavior.&#xA;- cli/cmd/root.go:31-39 initializes the root command and calls `rootCmd.Execute()` instead of `ExecuteContext(ctx)`.&#xA;- cli/cmd/work.go:37-40 advertises `signal` as a stop condition, but no production signal handler cancels `cmd.Context()`.&#xA;- docs/helix/01-frame/features/FEAT-010-task-execution.md:423-434 and docs/helix/02-design/solution-designs/SD-025-task-execution-lifecycle.md:177-190 require first-signal cooperative shutdown and the exact operator message `Cancel received, shutting down gracefully`.&#xA;&#xA;PROPOSED FIX&#xA;- Install a root SIGINT/SIGTERM handler for production CLI execution using `signal.NotifyContext` or equivalent.&#xA;- Execute Cobra with the signal-backed context.&#xA;- On first signal, print `Cancel received, shutting down gracefully` exactly once to the operator-visible stream before cancellation-dependent cleanup begins.&#xA;- On a second signal, hard-abort or restore default signal behavior so wedged cleanup can still be killed.&#xA;- Keep tests deterministic by exposing a small helper/seam rather than sending real OS signals where unit tests can use a cancellable context.&#xA;&#xA;NON-SCOPE&#xA;- Do not decide bead cleanup policy here beyond propagating context cancellation.&#xA;- Do not change server worker stop/watchdog behavior.
