    <title>worker: mirror live attempt liveness without tracker heartbeat spam</title>
    <description>
PROBLEM&#xA;During a long `ddx work` attempt, operators can see the bead stuck at `status=in_progress` with `extra.execute-loop-heartbeat-at` still at claim time even while the child agent is actively emitting progress. That is misleading: the tracker field is a durable claim marker, not a high-frequency liveness stream. Spamming `.ddx/beads.jsonl` with heartbeat rewrites would create tracker churn, auto-commit noise, and git/index contention. Operators need to answer &#34;is this worker alive and what is it doing?&#34; from worker logs/status and, when a server is reachable, from best-effort worker call-home events.&#xA;&#xA;ROOT CAUSE WITH FILE:LINE&#xA;- `cli/internal/bead/store.go:901-925` writes `extra.execute-loop-heartbeat-at` only on claim, then records ongoing claim liveness via `TouchClaimHeartbeat` outside the tracker. That is correct for avoiding beads.jsonl churn, but the operator-facing bead row therefore does not show live progress.&#xA;- `cli/internal/agent/work/heartbeat.go:13-35` refreshes only the external claim lease. It has no liveness reporter hook for worker status, worker logs, or server call-home.&#xA;- `cli/internal/agent/execute_bead_loop.go:1477-1491` wraps the attempt in `work.WithHeartbeat`, but the periodic tick does not emit a structured worker heartbeat/progress event.&#xA;- `cli/cmd/execute_loop_shared.go:164-183` wires `ddx work` to `workerprobe.TeeJSONL` over the loop EventSink only. `cli/internal/agent/workerprobe/probe.go:456-509` mirrors only newline-delimited loop events, so a long attempt with no loop event until `bead.result` leaves the server view stale.&#xA;- `cli/internal/server/worker_ingest.go:110-120` updates server `last_event_at` only when `/api/workers/&lt;id&gt;/event` receives an event, so the server cannot distinguish active long-running work from a silent/stale worker.&#xA;&#xA;PROPOSED FIX&#xA;Keep `.ddx/beads.jsonl` low-frequency and durable: claim, terminal outcome, and explicit lifecycle events only. Add a worker liveness path that runs with `ddx work` and `ddx try`: on each claim heartbeat tick, update a local worker/attempt status sidecar or log-derived status under `.ddx/workers/&lt;worker-id&gt;/status.json` (or the existing worker status path if available) with `worker_id`, `current_bead`, `attempt_id`, `phase`, `last_activity_at`, `route`/policy summary, and child/worker PID when available. When `workerprobe` is connected, also mirror a lightweight `worker.heartbeat` or `attempt.heartbeat` event to `/api/workers/&lt;id&gt;/event` so `last_event_at` stays fresh. Server unavailability must fail open and should only be logged locally.&#xA;&#xA;NON-SCOPE&#xA;- Do not reintroduce frequent `beads.jsonl` heartbeat rewrites or heartbeat auto-commits.&#xA;- Do not make the server authoritative for worker claims, scheduling, or cancellation. ADR-022 autonomous worker semantics remain intact.&#xA;- Do not rename existing worker ingest endpoints unless a separate contract bead is filed.&#xA;&#xA;Parent: none. Dependencies: none; this can be implemented after the P0 routing fix but does not block it.
    </description>
