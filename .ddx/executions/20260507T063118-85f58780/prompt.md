    <title>work: serialize model-backed readiness before claim races</title>
    <description>
PROBLEM&#xA;`ddx work` now runs model-backed bead readiness intake and bead-lifecycle lint after selecting a ready bead but before atomically claiming it. With two workers on the same project, both workers can select the same bead, both spend model calls on intake/lint, and both can append pre-dispatch events before one loses `Store.Claim`. If intake returns `actionable_but_rewritten`, the loser can also mutate bead description/acceptance before owning the bead. Atomic `Store.Claim` still prevents duplicate implementation, but it no longer protects the expensive and stateful pre-claim phase.&#xA;&#xA;ROOT CAUSE WITH FILE:LINE&#xA;- cli/internal/agent/execute_bead_loop.go:686-765 calls `runtime.PreClaimIntakeHook` and handles rewrite/block outcomes before `w.Store.Claim` is attempted.&#xA;- cli/internal/agent/execute_bead_loop.go:768-807 calls `runtime.PreDispatchLintHook` and `appendPreDispatchLintEvent` before the claim, so concurrent workers can append duplicate lint events for the same open bead.&#xA;- cli/internal/agent/execute_bead_loop.go:809-823 only detects the ownership race after intake/lint have already run.&#xA;- cli/internal/agent/preclaim_intake_rewrite.go:39-58
