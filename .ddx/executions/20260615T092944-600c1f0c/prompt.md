    <title>agent: enforce request-timeout as provider wall-clock cap</title>
    <description>
PROBLEM&#xA;During dogfood of ddx-3b721804 on 2026-06-13, the worker was started with `ddx work --once --request-timeout 25m`, but the primary Claude provider process stayed alive for roughly 56 minutes while tool activity and watcher stalls continued. The operator expectation for `--request-timeout` is a bounded provider/attempt wall-clock cap with cleanup. DDx currently allowed the provider leg to run more than twice the configured timeout, which makes unattended burn-down unreliable.&#xA;&#xA;ROOT CAUSE WITH FILE:LINE&#xA;cli/internal/agent/serviceconfig.go:24-33 documents `DefaultProviderRequestTimeout` and says `--request-timeout` applies to execute-bead/work one-off debugging.&#xA;cli/internal/agent/agent_runner_service.go:112-125 resolves and forwards `ProviderTimeout` into the service request.&#xA;cli/internal/agent/agent_runner_service.go:337-366 resets idle/tool-call timers on service events, but there is no absolute wall-clock timer that cancels a provider session that remains active through endless tool events.&#xA;cli/internal/server/workers.go:734-787 forwards server worker request timeout into loop overrides, so server-managed workers will inherit the same ambiguity unless the boundary is made explicit and tested.&#xA;&#xA;PROPOSED FIX&#xA;Make `--request-timeout` an absolute provider-session wall-clock cap for execute-bead/work attempts, distinct from idle-read and per-tool-call timers. On expiry, cancel the provider context, reap the attempt process tree, release/park the lease according to existing timeout policy, and write durable evidence naming configured_timeout, elapsed, phase, provider PID, and cleanup result.&#xA;&#xA;NON-SCOPE&#xA;Do not remove idle-read timeout or tool-call timeout. Do not call real providers in tests. Do not make long-running project tests impossible; if a longer cap is needed, the operator can set one explicitly.
    </description>
