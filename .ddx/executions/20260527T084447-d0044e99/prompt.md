    <title>execute-bead: set Pdeathsig on harness children so the kernel kills them when the worker dies abnormally (SIGKILL/crash)</title>
    <description>
PROBLEM&#xA;When a `ddx work` orchestrator dies abnormally (SIGKILL, crash, OOM, host reboot of the supervising process) while a bead attempt is in flight, the spawned harness child (e.g. `claude --print -p --verbose --output-format stream-json ... &lt;execute-bead&gt;` or `codex exec --json ...`) is NOT reaped. It reparents to PID 1 and keeps running indefinitely. Observed 2026-05-27 on the dev host: a `claude --print` execute-bead child orphaned with PPID 1 for ~9h53m after its `ddx work` parent had gone. Graceful SIGINT/SIGTERM cleanup works, but it cannot run when the parent is killed uncatchably.&#xA;&#xA;ROOT CAUSE (file:line)&#xA;- cli/internal/agent/executor.go:156-157 — OSExecutor.ExecuteInDir spawns the harness via exec.Command(binary, args...) then calls cmdSetProcessGroup(cmd).&#xA;- cli/internal/agent/executor_unix.go:13-15 — cmdSetProcessGroup sets ONLY SysProcAttr{Setpgid: true}. There is NO Pdeathsig, so the kernel does nothing automatically when the parent dies.&#xA;- cli/internal/agent/executor.go:190-194 / :231-255 — cmdKillProcessGroup is only invoked from the executor&#39;s graceful paths (idle/wall-clock timeout, auth/rate-limit pattern, ctx cancellation). These never run on an uncatchable parent death.&#xA;- cli/cmd/root.go:49-50 — the worker context is signal.NotifyContext on os.Interrupt + SIGTERM only; SIGKILL/crash cannot be caught, so the defer/stopProcess chain never fires and the child orphans.&#xA;- cli/internal/agent/executor_windows.go:7-8 — cmdSetProcessGroup is a documented no-op on Windows; Pdeathsig is Linux-only, so the non-Linux gap must be handled explicitly.&#xA;&#xA;PROPOSED FIX&#xA;Set SysProcAttr.Pdeathsig = SIGKILL in cmdSetProcessGroup (cli/internal/agent/executor_unix.go) so the kernel signals the harness child when the parent thread dies, regardless of how the parent died. Handle the well-known caveat that Pdeathsig keys off the spawning OS *thread*: guard against Go runtime thread migration (runtime.LockOSThread around cmd.Start, or re-verify parent PID in the child path) so the death signal actually fires. Document the macOS/Windows gap (Pdeathsig is Linux-only) and the chosen fallback there (the orphan reaper child bead is the cross-platform backstop). Note: Pdeathsig is in golang.org/x/sys/unix / syscall on Linux only; keep the !windows build constraint and any darwin-specific behavior correct.&#xA;&#xA;NON-SCOPE&#xA;- Graceful SIGINT/SIGTERM shutdown already works (executor.go:231-255); this change must NOT regress it.&#xA;- Startup-time reaping of orphans from already-dead prior workers is a separate child (the backstop reaper).&#xA;- docker-clone backend process-group cleanup is a separate child.&#xA;- The harness-runner-INTERNAL leak is covered by fizeau-50b6f055; this bead is strictly the ddx orchestrator -&gt; harness-child spawn boundary.&#xA;&#xA;CROSS-REF: fizeau-50b6f055 (fizeau harness-runner process-group cleanup).
    </description>
