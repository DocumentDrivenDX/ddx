# Why this attempt declined to commit

The bead asks for an end-to-end migration of the server-spawn path from in-process execution to subprocess `exec ddx work`. Six tightly-coupled ACs touch the most intricate file in the server package.

## Actual surface (vs stated estimate)

`cli/internal/server/workers.go` is 1,973 lines. The current `runWorker` (lines 658-940, ~285 lines) is a dense in-process orchestrator that knits together:

- in-process `agent.ExecuteBeadWorker` construction (lines 811-815)
- per-project `LandCoordinator` single-flight serialization (line 676; `m.LandCoordinators.Get(projectRoot)`)
- in-process post-merge `DefaultBeadReviewer` (lines 801-809)
- in-process `escalation.CostCapTracker` (lines 753-781) — per-worker billed cost accumulation
- in-process `RoutePreflight` typed-incompatibility gate via FizeauService (line 838 → `buildRoutePreflight` lines 622-649)
- per-attempt `singleTierAttempt` closure with config resolution + gate evaluation + leftover-evidence safety net (lines 679-747)
- progress channel piped to SSE subscribers via `drainProgress` (lines 421, 440, 1374)
- watchdog phase-stall tracking via shared `workerHandle.lastPhaseTS`
- terminal-state preservation (stopped / reaped) merged with loop result (lines 860-936)
- `BeadWorkerFactory` injection point for test fakes (workers.go:213, used by 7 sibling test files)

Replacing the in-process call with `exec ddx work` collapses all of those in-process couplings — they now run inside the child process — but the parent server still needs equivalent observability/control via:

- `/api/workers/{id}/event` ingestion (already exists from step 1, server.go:884) for progress
- PID + signal escalation for watchdog/Stop (PID field exists at workers.go:147; signal plumbing is partial in workers_reap_unix.go / workers_reap_windows.go)
- worker-events.jsonl tail or HTTP-mirrored events for SSE bridging
- Cross-process Stop / wakeCh equivalent (currently the in-process wakeCh at workers.go:203 has no subprocess analogue)

## Test surface

`cli/internal/server/workers_test.go` alone is 1,107 lines. It and 6 sibling files use `BeadWorkerFactory` to inject a fake in-process executor:

- workers_test.go
- workers_stop_propagation_test.go
- workers_route_preflight_test.go
- workers_e2e_test.go
- workers_config_test.go
- graphql_adapters_default_profile_test.go

With subprocess execution, `BeadWorkerFactory` is meaningless — the parent has no executor to inject; the child process owns it. Those tests need to be replaced by either:

- a built-in subprocess-stub binary (e.g. `go test -test.run=...&ddx-fake-work-stub` reentry pattern) that the parent execs in tests, or
- deletion as no-longer-applicable (semantics now belong to `ddx work`'s own test surface in cli/cmd/agent_execute_loop_*_test.go).

Plus 5 additional files use `ExecuteLoopWorkerSpec` directly:

- workers_watchdog_test.go
- workers_stop_test.go
- workers_prune_test.go
- graphql_backfill_test.go
- graphql_adapters.go (production code — adapter for GraphQL StartWorker mutation)

That's 12 files referencing structures that AC #3 demands be deleted. This is the bulk of the work, not "test updates."

## AC #5 (real-harness end-to-end)

> server-spawned worker (via the new exec path) registers via the new endpoints AND successfully invokes the claude harness (real claude binary, not a stub) on a fixture bead AND completes the attempt with a commit

Even with the fall-back to a scripted-claude-stub, this requires:
- a clean fixture repo with a real ready bead (uses `ddx-50da9674` clean fixture, currently in flight)
- a stub or real claude binary on `$PATH` inside the test harness
- a server bound to a free port
- the new exec path wired in
- the worker registering against the running server's `/api/workers/register`
- an end-to-end commit landing through the `LandCoordinator` (which has moved out of the parent process into the child).

That is a substantial new integration test — comparable to the existing `workers_e2e_test.go` suite — distinct from the refactor proper.

Title: `ADR-022 step 7a: workers.go runWorker → exec ddx work subprocess (parent-side plumbing)`

Scope:
