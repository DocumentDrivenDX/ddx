    <title>execute-bead: worker exits on transient index.lock contention during durable-audit commit instead of retrying — kills unattended multi-worker drain</title>
    <description>
PROBLEM&#xA;Under multi-worker operation, a ddx work worker EXITS with &#34;operator attention required: DDx could not commit durable audit outputs; resolve the git failure before continuing autonomous work&#34; when the durable-audit/tracker commit hits transient .git/index.lock contention. Observed 2026-05-27: with 2 ddx workers + a fizeau worker + the server sharing one project .git, a worker died on `git add` tracker -&gt; &#34;.git/index.lock: File exists ... no owner found but lock is fresh (age 19s &lt; 30s)&#34;. A transient lock conflict stops the whole worker; the more workers, the more often this fires, directly defeating unattended steady-state drain.&#xA;&#xA;ROOT CAUSE WITH FILE:LINE&#xA;- git .git/index.lock carries no pid, so gitlock.IndexLockOwner finds no owner. RecoverGitIndexLock (cli/internal/gitlock/gitlock.go:144-158) only reaps an ownerless lock once age &gt;= StaleAge (30s, gitlock.go:26); below that it returns Removed=false (&#34;no owner found but lock is fresh&#34;).&#xA;- RunGitWithIndexLockRecovery retries only RecoveryAttempts=3 (gitlock.go:30) with LiveOwnerWait=500ms (gitlock.go:34) ~= 1.5s total, far short of the 30s reap floor, then returns the index.lock error.&#xA;- commitTrackerLocked (cli/internal/agent/execute_bead.go:2153) calls runGitWithIndexLockRecovery for `git add`; its error becomes &#34;staging tracker: ...&#34;. That propagates to the durable-audit-commit exit at cli/internal/agent/execute_bead_loop.go:1175, which stops the worker (operator attention) instead of treating transient lock contention as retryable.&#xA;&#xA;PROPOSED FIX&#xA;1. Do NOT exit the drain on transient .git/index.lock contention during the durable-audit/tracker commit. Classify index.lock contention as transient (mirroring the pre-claim tracker-contention class from ddx-df77e668), release the bead, and continue the loop so the next iteration retries. A transient lock must never stop a worker.&#xA;2. AND/OR raise the recovery patience for tracker/audit commits to a bounded backoff that can outlast typical contention (up to ~the StaleAge window) before failing.&#xA;3. Optional: reap a provably-orphaned lock sooner (e.g. confirm no live git holds the repo via lsof) while keeping the 30s floor when liveness is unknown.&#xA;Recommend (1) as the primary fix: transient lock contention is retryable, not operator-attention-worthy.&#xA;&#xA;NON-SCOPE&#xA;- Do not weaken safety against removing a live git process index.lock.&#xA;- Do not change the landing-path read-tree retry (already has backoff, ddx-7e659c95).&#xA;&#xA;CROSS-REF: same multi-worker-resilience theme as ddx-57c40485 (lock scoping) and ddx-df77e668 (pre-claim tracker-contention). This is the post-fix gap on the COMMIT side.
    </description>
