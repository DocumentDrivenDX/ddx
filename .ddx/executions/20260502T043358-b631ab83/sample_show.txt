Title:    bug: script test-harness leaks into production tier fallback chain
Type:     bug
Status:   closed
Priority: 1
Labels:   kind:bugfix, area:agent, area:routing
Owner:    erik
Desc:     ## Problem

The `script` harness is a test-only sentinel (`cli/internal/agent/registry.go:127` — comment says "Binary: ddx-script-agent // sentinel — never actually exec'd") that interprets directive files for deterministic test execution. It has no LLM and cannot produce real work.

Despite being test-only, it shows up in production tier fallback chains. Observed 2026-04-17 on bead agent-ecc79bf0:

```
22:28:39  tier=cheap      harness=lmstudio  model=qwen3.5-27b    execution_failed (exec: no command)
22:28:43  tier=standard   harness=script    model=<prompt.md>    execution_failed (unknown directive "<execute-bead>")
22:28:47  tier=smart      harness=codex     model=gpt-5.4        (succeeded)
```

The `<execute-bead>` XML prompt that execute-bead emits is obviously not a valid script directive file — script's parser choked on it on the first line.

## Root cause

`cli/internal/agent/registry.go:125-132` registers `script` with:
- `CostClass: "local"`
- `IsLocal: true`

`cli/internal/agent/routing.go:scoreCandidate` rewards any `local` harness with +25 (standard profile) or +40 (cheap profile). Because script is registered as local AND always "available" (it doesn't need an LLM or network), it scores highly whenever real local harnesses (LM Studio, ollama, embedded agent) are missing or failed.

There is no filter in the routing logic that excludes `script` (or `virtual`) from execute-bead candidate selection.

## Expected behavior

`script` and `virtual` should be test-only harnesses, never selected by execute-bead / execute-loop production routing. They should only be reachable via an explicit `--harness script` / `--harness virtual` override, not via tier-based profile routing.

## Proposed fix

1. Add a `TestOnly bool` field on the registry struct (or reuse an existing negative marker).
2. Filter test-only harnesses out of the candidate list in routing.go before scoring.
3. Keep explicit opt-in (`--harness script`) working so the existing script tests still pass.
4. Add a regression test that execute-loop never selects script/virtual when real harnesses are unavailable.

## Files likely touched
- cli/internal/agent/registry.go — add TestOnly flag
- cli/internal/agent/routing.go — filter TestOnly before scoring
- cli/internal/agent/routing_test.go — regression test
Accept:   execute-loop never selects `script` or `virtual` harness via tier routing (cheap/standard/smart). Explicit `--harness script` still works. Regression test covers the fallback-when-real-harnesses-fail case.
