    <title>Add chat interface to project home page: prompt ddx remotely without the TUI</title>
    <description>
## Observed / Motivation&#xA;&#xA;User wants to &#34;lift work out of the TUI entirely and allow non-tmux users to work remotely.&#34; Today the only way to prompt the DDx agent stack interactively is a terminal session. The web UI is read-mostly: dispatch three actions (drain/align/checks) and review results. There is no surface that lets an operator say &#34;go do X&#34; against the project in natural language from a browser.&#xA;&#xA;Related UX: the project home page already exists (`/nodes/.../projects/&lt;id&gt;/`), renders queue summary + actions, and is the natural host for a chat affordance. First the home page needs to be reachable from the sidebar (`ddx-2513f08c`) — this bead depends on that work.&#xA;&#xA;## The tension&#xA;&#xA;The DDx harness abstraction (`cli/internal/agent/service_run.go`, `service.Execute`) is designed for **non-interactive** use: prompt in, structured result out, optional streaming. This is the right shape for queue-draining, quorum, evaluation, and automation. It is **not** the shape of a TUI conversation with tab completion, mid-run steering, and back-channel signals.&#xA;&#xA;An operator-facing chat can land on either side of that line. The bead records all four options so the design phase can pick one deliberately.&#xA;&#xA;## Options (design phase decides, recommended path noted)&#xA;&#xA;### Option A — Chat over the harness abstraction (RECOMMENDED starting point)&#xA;&#xA;- Chat UI posts a prompt → server dispatches a session via `service.Execute` (or a wrapper) → streams tool calls + response chunks back over a GraphQL subscription.&#xA;- Message history is persisted as a thread; each user turn = one new session. Responses reuse the existing `recentEvents` stream shape already used on the workers detail page.&#xA;- Works with every harness today because the harness abstraction is what it is. claude, codex, agent (embedded) all participate.&#xA;- **Trade-off:** no mid-run steering, no tab completion, no live back-and-forth inside one agent invocation.&#xA;- Matches the user&#39;s stated target audience (non-TUI users who want remote browser access). Those users aren&#39;t trying to steer a TUI — they&#39;re trying to leave it behind.&#xA;&#xA;### Option B — PTY gateway per harness&#xA;&#xA;- Server owns a PTY pool. Each chat thread attaches to one harness process (claude/codex/etc.) over a WebSocket; browser renders xterm.js.&#xA;- Full interactivity, including TUI steering.&#xA;- **Trade-off:** PTY lifecycle, auth, resource limits, tab-close-loses-session semantics, multi-user access control — all real. The harness abstraction is bypassed for this mode, which creates a second execution path to maintain.&#xA;&#xA;### Option C — Hybrid (chat-first + pop-out terminal)&#xA;&#xA;- Option A is the primary surface. When a chat reaches a point that needs interactive steering, a &#34;Open terminal&#34; escape hatch attaches a PTY to the current harness session.&#xA;- **Trade-off:** more surface area, but honest about both use cases. Most sessions stay on the cheap path.&#xA;&#xA;### Option D — Web-tmux / ttyd wrapper&#xA;&#xA;- Embed the existing terminal tooling directly: ttyd or xterm.js over a tmux socket owned by the server.&#xA;- **Trade-off:** fastest to ship, zero harness work, but the UX is a terminal in a browser — not &#34;chat.&#34; Requires tmux. Doesn&#39;t serve the stated &#34;non-tmux users&#34; audience.&#xA;&#xA;### Recommendation&#xA;&#xA;Start with **A**. If the design note documents real interactive-steering needs we can&#39;t paper over, the right follow-up is **C** (A + PTY escape hatch), not B (PTY-only).&#xA;&#xA;## Scope of this bead&#xA;&#xA;- Design note in `docs/helix/02-design/` locking the option, the message schema, persistence model, and auth posture. Implementation PR links to it.&#xA;- First implementation pass: Option A.&#xA;- Home page gains a chat pane as its primary interaction surface, alongside the existing queue summary + actions (which stay — they&#39;re quick-dispatch affordances, not replaced by chat).&#xA;&#xA;## Subsidiary questions the design note must answer&#xA;&#xA;1. **Message persistence.** Are chat threads per-project or per-user? Stored where (`.ddx/chat/&lt;thread-id&gt;/` or elsewhere)? Is history shared across operators on the same project, or isolated per operator identity?&#xA;2. **Harness selection.** Does the chat pick a harness per-thread (picker in the UI) or per-message (re-route each turn)? Default to profile-based selection?&#xA;3. **Cost accountability.** Every chat message spawns a session. Sessions are already tracked in the sessions index (`ddx-2ceb02fa`). The chat thread should surface cumulative cost to the operator so they don&#39;t blow budget unknowingly.&#xA;4. **Tool execution scope.** Chat prompts will trigger agent tool calls that read/write the project tree. Is the chat session bounded to the same permissions as `ddx agent run`? Does it run in the project worktree, a worktree per thread, a scratch dir? Default safe answer: same worktree as `ddx agent run` (user trusts ddx running anyway), but design note records it.&#xA;5. **Concurrency.** What happens if two operators have the chat open on the same project and both send messages? Default: independent threads, each with its own session — no shared state.&#xA;6. **Auth.** Today the server assumes single-operator localhost. Remote use implies some identity concept. Design note picks a posture: delegate to reverse proxy, ship a simple token, or punt and require tsnet + single-operator for v1.&#xA;&#xA;## Out of scope&#xA;&#xA;- Full PTY implementation (Option B). Follow-up bead if A+hybrid doesn&#39;t suffice.&#xA;- Cross-project chat.&#xA;- Voice input.
    </description>
    <acceptance>
**User story:** As an operator using a browser — on a machine without a terminal, or a phone, or a laptop away from my dev box — I can open my project&#39;s home page, type a prompt into a chat pane, and see the DDx agent work on it in real time. Tool calls stream as they happen, costs accrue visibly, and the conversation persists so I can come back to it.&#xA;&#xA;**Acceptance criteria (written for Option A; rewrite if design note chooses B/C/D):**&#xA;&#xA;1. **Design note landed.** One-pager in `docs/helix/02-design/` records the chosen option, the answers to the six subsidiary questions, and the message/thread schema. Implementation PR links to it.&#xA;&#xA;2. **Thread model.** Chat threads persist under `.ddx/chat/&lt;thread-id&gt;/{manifest.json,messages.jsonl}`. One thread per project per operator by default; operators can create named threads. Messages.jsonl is append-only, one JSON object per message (user or assistant), with `sessionId` linking each assistant turn back to an entry in the sessions index (`ddx-2ceb02fa`).&#xA;&#xA;3. **GraphQL surface.** New resolvers:&#xA;   - Query: `chatThreads(projectId)`, `chatThread(id)`, `chatMessages(threadId, since?)`.&#xA;   - Mutation: `chatThreadCreate(projectId, title?)`, `chatSend(threadId, message, harness?, profile?)` → kicks off a session via `service.Execute`.&#xA;   - Subscription: `chatStream(threadId)` → emits assistant message chunks + tool call events (same event vocabulary as `workers.recentEvents`).&#xA;&#xA;4. **Home page UI.** Project home page gains a chat pane below the actions section. Pane shows thread list, active thread messages, input box, harness/profile picker. Tool calls render inline within assistant messages. Streaming updates as chunks arrive.&#xA;&#xA;5. **Session linkage.** Each assistant turn creates one session in the sessions index. Clicking an assistant message shows the session id and a link to the session detail (`/sessions/&lt;id&gt;`). Chat thread header shows cumulative session count + total cost for the thread.&#xA;&#xA;6. **Concurrency.** Two browsers open on the same thread see each other&#39;s messages within 2s (subscription-driven). Two operators on different threads in the same project do not interfere.&#xA;&#xA;7. **Interrupt.** The in-flight assistant turn can be cancelled via a Stop button; server sends the harness the equivalent of `ddx agent workers stop &lt;id&gt;` for the in-flight session. The thread records a cancelled turn, does not try to auto-resume.&#xA;&#xA;8. **Cost guardrail (soft).** If the cumulative thread cost passes a configurable ceiling (`.ddx/config.yaml: chat.cost_ceiling_usd`, default unset), the UI warns but does not block. Hard block is a follow-up.&#xA;&#xA;9. **Auth posture.**&#xA;   - Per design note. If the chosen posture is &#34;single-operator localhost / tsnet only for v1,&#34; the chat routes are served unauthenticated behind tsnet and the design note records the gap to close before broader deployment.&#xA;   - A small middleware gate is in place even for v1 so adding auth later is schema-local, not route-local.&#xA;&#xA;10. **Playwright e2e.**&#xA;    - Seeds a project with zero chat threads.&#xA;    - Navigates to project home, asserts the chat pane renders with &#34;Start a new thread&#34; affordance.&#xA;    - Starts a thread, sends a short prompt, simulates a streamed session response with one tool call, asserts chunks appear in order and the session id link is present.&#xA;    - Opens a second browser context on the same thread, asserts both see new messages within 2s.&#xA;    - Cancels an in-flight turn, asserts the cancelled state is persisted.&#xA;&#xA;11. **Depends on.**&#xA;    - `ddx-2513f08c` (home page reachable from sidebar) — hard dependency.&#xA;    - `ddx-2ceb02fa` (sessions index) — hard dependency for the session linkage + cost rollup.&#xA;&#xA;12. **Explicitly deferred.** PTY integration (Option B / pop-out terminal from Option C) is its own bead, filed only if the Option A rollout surfaces real interactive-steering needs.
    </acceptance>
