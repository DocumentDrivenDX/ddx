PROBLEM&#xA;The ddx-validate pre-commit hook runs ddx doctor on every commit that touches .ddx.yml, templates/, patterns/, or prompts/. It rejects the commit if ddx doctor exits non-zero. Today this blocked a 1121-file rename commit because ddx doctor was flagging unrelated legacy skill symlinks under .agents/skills/helix and .claude/skills/helix — which were not part of the staged change. The operator had to remove the symlinks (or know about the doctor issue) to unblock an entirely unrelated commit. The hook scope is too coarse: it validates GLOBAL repo health, not the staged change.&#xA;&#xA;ROOT CAUSE&#xA;lefthook.yml ddx-validate hook (see scripts/) runs `ddx doctor` with no scoping. ddx doctor checks installation health, plugin layout, git config, skill symlinks, sync status, etc. — all repo-global, none of which are mutated by an arbitrary staged change touching .ddx.yml or templates. A skill-symlink issue (or any other doctor finding unrelated to the staged files) thus blocks unrelated commits.&#xA;&#xA;PROPOSED FIX&#xA;- Replace the global `ddx doctor` invocation with a scoped validation that only checks the categories of state actually mutated by the staged change:&#xA;  - If .ddx.yml staged → run config-only validation (parse + schema).&#xA;  - If templates/, patterns/, prompts/ staged → run resource-only validation (the staged file&#39;s syntax/schema).&#xA;- Add a ddx doctor --scope=staged or ddx doctor --paths &lt;file&gt;... flag that takes a path list and only runs checks whose subject is in that list. Hook passes the staged path list.&#xA;- Keep the full doctor available as an operator command; just don&#39;t run it on every commit.&#xA;- Document the new scoping model in scripts/ or AGENTS.md.&#xA;&#xA;NON-SCOPE&#xA;- Do not change what ddx doctor checks at the full-run level; only scope it for hook usage.&#xA;- Do not change other lefthook hooks.&#xA;&#xA;DEPS&#xA;None.
