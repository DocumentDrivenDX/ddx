    <title>agent: release worker claims atomically on failure paths</title>
    <description>
PROBLEM&#xA;Some worker failure paths still call `Store.Unclaim`, which updates the bead row and then removes the claim heartbeat outside the same critical section. If heartbeat removal fails, `ReadyExecution` can keep excluding an open bead until the sidecar lease TTL expires, reducing multi-worker throughput and making failures look like queue starvation.&#xA;&#xA;ROOT CAUSE&#xA;- cli/internal/bead/store.go:1081-1100 implements `Unclaim` as row update followed by separate heartbeat removal.&#xA;- cli/internal/bead/store.go:1103-1130 implements `Release`, which atomically updates the row and clears the heartbeat under one lock.&#xA;- cli/internal/agent/execute_bead_loop.go:2365-2368 still uses `w.Store.Unclaim(candidate.ID)` on a worker refresh failure path; other worker paths should be audited for the same pattern.&#xA;&#xA;PROPOSED FIX&#xA;- Audit worker-owned failure/unclaim paths and replace them with `Release` where the current worker owns the claim.&#xA;- Add a fault-injection test that makes heartbeat removal fail and proves the bead is not left with a fresh sidecar lease after a worker failure path releases it.&#xA;- Preserve CLI/operator `ddx bead update --unclaim` semantics where direct `Unclaim` is still appropriate.&#xA;&#xA;NON-SCOPE&#xA;- Do not change claim ownership rules or sidecar lease TTLs in this bead.&#xA;- Do not close beads on failure; failed attempts must remain open/unclaimed unless existing outcome policy says otherwise.
