    <title>execute-bead: reap orphaned harness children on `ddx work` startup (backstop for dead prior workers)</title>
    <description>
PROBLEM&#xA;If a `ddx work` orchestrator already died abnormally before any kernel parent-death signal could apply (e.g. host reboot, pre-Pdeathsig binary, or a child whose Pdeathsig misfired), its spawned harness child (e.g. `claude --print ... &lt;execute-bead&gt;` or `codex exec --json ...`) is left running as a PID-1 orphan, burning a harness session and anchoring leaked subprocess trees. Observed 2026-05-27: a `claude --print` execute-bead child orphaned with PPID 1 for ~9h53m after its parent worker had gone. Nothing in the startup path adopts or kills harness children whose owning worker PID is gone.&#xA;&#xA;ROOT CAUSE (file:line)&#xA;- cli/internal/agent/executor.go:190-194 / :231-255 — harness-child cleanup (cmdKillProcessGroup) is invoked ONLY from the live executor&#39;s graceful paths. A dead parent runs none of them, so its orphan persists with no later sweep.&#xA;- cli/cmd/root.go:49-50 — worker context catches only os.Interrupt + SIGTERM; SIGKILL/crash leave no cleanup hook, and there is no startup reaper to compensate.&#xA;- There is no code path on `ddx work` startup that scans for execute-bead harness processes whose parent PID is gone (reparented to PID 1) or whose recorded owner PID no longer exists.&#xA;&#xA;PROPOSED FIX&#xA;On `ddx work` startup (at register/claim time), reap orphaned execute-bead harness children left by prior dead workers: scan for harness processes (matching the execute-bead invocation signature / recorded owner PID) whose parent is PID 1 or whose recorded owner PID no longer exists, and SIGKILL their process groups. Scope the sweep strictly to the current project/workspace (e.g. via the project root / workspace marker already recorded for attempts) so it never kills harness children of OTHER live workers or unrelated processes. Tie into the existing lease-release / operator-attention machinery (cross-ref ddx-8f2e0ebf) for release of any stale lease the orphan was holding.&#xA;&#xA;NON-SCOPE&#xA;- The kernel-enforced Pdeathsig change in cmdSetProcessGroup is a separate child (the primary fix); this reaper is the cross-platform backstop and must not depend on Pdeathsig being present.&#xA;- Graceful SIGINT/SIGTERM shutdown of live workers (executor.go:231-255) is out of scope and must not be touched.&#xA;- docker-clone backend cleanup is a separate child.&#xA;- The harness-runner-INTERNAL leak is covered by fizeau-50b6f055.&#xA;&#xA;CROSS-REF: ddx-8f2e0ebf (lease release / operator-attention), fizeau-50b6f055.
    </description>
