# PROBLEM&#xA;Federated nodes share bead queues through git, but nothing in the server-managed worker path ever runs fetch/merge/push — queues silently diverge. Observed live 2026-07-05/06:&#xA;- eitri&#39;s ddx checkout opened the session at main ahead 365 / behind 38 vs origin (weeks of worker closes never pushed; the retired /tmp factory-supervisor stopgap used to do per-cycle pull+push, server-managed workers do not).&#xA;- sindri&#39;s ddx checkout (queried via its server API /api/beads/status) reports total=2315 vs eitri&#39;s 2339 — it is missing 24 beads created on eitri including the P0 fixes ddx-57faa17c and ddx-a6e99ad3, and it holds 1 closed bead (closed=2206 vs 2205) never pushed to origin.&#xA;Latent failure: FEAT-026 partitions queue work by project owner, but ownership relies on both nodes seeing the same tracker. A worker started against a stale checkout re-executes already-closed beads or conflicts with landed work. FEAT-026&#39;s frame explicitly lists &#39;bead queues must be reconciled by hand&#39; as the gap this bead closes.&#xA;&#xA;# ROOT CAUSE&#xA;No component owns tracker git sync during autonomous drains. cli/cmd/execute_loop_shared.go / the execute-bead loop land results to LOCAL main only; nothing fetches origin or pushes after close. The old stopgap supervisor script (per-cycle clean+pull/merge+push, see ddx-171dc478 stopgap) was retired when server-managed workers replaced it, dropping the sync side-effect.&#xA;&#xA;# DIRECTIVE (operator, 2026-07-06)&#xA;Force a git sync BEFORE and AFTER claiming beads. The claim boundary — not just close/idle — is the sync point:&#xA;1. PRE-CLAIM: git fetch origin + merge origin/main --no-ff before candidate selection, so the worker picks from the freshest queue and never claims a bead another node already closed or claimed.&#xA;2. POST-CLAIM: commit + push the claim immediately, publishing it so other nodes&#39; pre-claim syncs see the bead as taken. The cross-node double-execution window shrinks to the interval between one node&#39;s post-claim push and the other&#39;s pre-claim fetch.&#xA;3. POST-CLOSE/LAND: push the close and landed result (as before).&#xA;Idle-tick sync remains as a low-frequency backstop (default ~5min) for nodes that are not claiming.&#xA;&#xA;# PROPOSED FIX&#xA;Implement the three sync points above in the worker loop. Merge with --no-ff only (never squash/rebase; union-merge JSONL conflicts per existing tracker-merge conventions). Pre-claim sync failure (unreachable origin) degrades to claim-from-local with a warning event — availability over freshness, matching the worker-autonomy posture of ADR-022. Push failures after claim retry with backoff, then emit loop.operator_attention rather than blocking the drain. Respect LandCoordinator (skip merge while a land is in flight). Gate behind --tracker-sync (default ON for --watch), mirroring the --no-X discoverable-opt-out convention enforced by cli/cmd/flag_design_test.go.&#xA;&#xA;# NON-SCOPE&#xA;- Hub-forwarded owner-targeted writes (FEAT-026 story 14 beads ddx-3f3e1cbe, ddx-20e7ebf0, ddx-0793ac75).&#xA;- Real-time distributed claim arbitration beyond the git claim protocol above (ownership partitioning remains the primary collision guard).&#xA;- Fixing sindri&#39;s currently-stale checkout (operational one-time reconcile; this bead prevents recurrence).&#xA;&#xA;Related: FEAT-026, ADR-007-federation-topology, ddx-171dc478 (closed; safe-boundary precedent).&#xA;&#xA;No deps.
