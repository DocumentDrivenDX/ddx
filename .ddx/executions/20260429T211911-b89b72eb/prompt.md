    <title>pre-claim triage: complexity evaluator + auto-splitter for recursive epic decomposition before agent dispatch</title>
    <description>
## Goal&#xA;&#xA;Today, when execute-loop dispatches a coarse bead (an unsplit epic) to an agent, the agent often returns `no_changes` with reasoning like &#34;this is an epic, needs child-bead breakdown, not monolithic execution&#34; — wasting a worker attempt, leaving the bead in a half-owned claimed state, and forcing manual operator intervention.&#xA;&#xA;Concrete recent example: `agent-6c7a4c11` (Harness × Model Matrix Benchmark — initial tranche) on this user&#39;s `agent` repo was dispatched as a single bead, the agent declined to execute monolithically, the worker exited without unclaiming. The bead is now stuck claimed by a dead worker with no change applied.&#xA;&#xA;The fix is to run a triage step **before** Claim: evaluate whether the bead is atomic (one PR&#39;s worth of work, single AC focus) or decomposable (multiple distinct deliverables, AC items map to separable PRs). If decomposable, run a splitter that produces child beads (parent-linked, AC-distributed), and skip dispatch of the parent. Children themselves re-enter triage — recursive decomposition with a depth cap.&#xA;&#xA;## Concrete change&#xA;&#xA;Insert a `complexityGate` phase in `cli/internal/agent/execute_bead_loop.go` between bead-pick and Claim:&#xA;&#xA;```&#xA;next-candidate → ResolveRoute → preflight gate (ddx-98e6e9ef)&#xA;  → complexityGate (NEW)&#xA;  → if atomic:        Claim → invoke&#xA;  → if decomposable:  splitter → file children with parent link → mark parent blocked-by-children → re-pick&#xA;  → if ambiguous:     surface to human triage (event + label), don&#39;t claim&#xA;```&#xA;&#xA;Two prompts, both shipped with the default DDx plugin under `library/prompts/triage/`:&#xA;&#xA;1. **`complexity-eval.md`** — given bead body + AC + deps + history (prior attempts, reviewer BLOCKs), classify `{atomic, decomposable, ambiguous}` with reasoning + confidence.&#xA;2. **`bead-split.md`** — given a decomposable bead, emit N child beads (JSON) with: title, AC, parent link, spec-id inheritance, in-scope files subset, labels. Children must collectively cover the parent&#39;s AC.&#xA;&#xA;Both prompts use the embedded `agent` harness with profile=default (cheap tier, local-first) — the gate&#39;s cost/latency must be &lt;&lt; a wasted dispatch.&#xA;&#xA;## Historical evidence engineering&#xA;&#xA;Both prompts must be engineered against historical evidence, not vibes. Harvest a corpus from `.ddx/beads.jsonl` (this project + the user&#39;s `agent` project, which has 5 months of bead history including known epic-rejection cases):&#xA;&#xA;- Beads closed `no_changes` whose status text mentions &#34;epic&#34;, &#34;split&#34;, &#34;breakdown&#34;, &#34;scope&#34;, or &#34;monolithic&#34; → positive examples for **decomposable**.&#xA;- Beads with `attempt_count &gt;= 3` or repeated `kind:reopen` events from reviewer BLOCKs citing scope → positive examples for **decomposable**.&#xA;- Beads that were eventually split manually (parent epic + N child tasks filed in a single tracker batch) → use parent body as splitter input, the actual children as ground-truth output.&#xA;- Beads closed cleanly on first attempt with single-file or small-diff commits → positive examples for **atomic**.&#xA;&#xA;Hold out 20% as eval. Report `complexity-eval` accuracy/F1 on classification; report `bead-split` AC-coverage rate (children&#39;s AC tokens together cover parent&#39;s AC) and child-count distribution vs. ground-truth.&#xA;&#xA;The harvest script (`scripts/triage/harvest-corpus.go` or equivalent) is committed and reproducible.&#xA;&#xA;## Recursion + depth cap&#xA;&#xA;Children re-enter the gate. Default depth cap = 3, configurable via `agent.triage.max_decomposition_depth` in `.ddx/config.yaml`. At cap: surface to human (`kind:triage-overflow` event, `status=blocked`, `label=needs-human-decomposition`). Never silently dispatch a parent that hit cap.&#xA;&#xA;## In-scope files&#xA;&#xA;- `cli/internal/agent/execute_bead_loop.go` — new `complexityGate` phase pre-Claim&#xA;- `cli/internal/agent/triage.go` (new) — gate logic, prompt invocation, child filing&#xA;- `library/prompts/triage/complexity-eval.md` (new)&#xA;- `library/prompts/triage/bead-split.md` (new)&#xA;- `cli/internal/agent/triage_test.go` (new) — unit tests against held-out historical corpus&#xA;- `library/prompts/triage/eval-corpus.jsonl` (new) — frozen eval slice (training slice can be regenerated)&#xA;- `scripts/triage/harvest-corpus.go` (new) — corpus regeneration&#xA;- `docs/triage/decomposition.md` (new) — operator docs&#xA;&#xA;## Out of scope (file follow-ups if/when needed)&#xA;&#xA;- Backfill triage on already-claimed in-flight beads (e.g. unsticking agent-6c7a4c11 retroactively) — operational bead, not part of this feature.&#xA;- A web UI for visualizing the triage decision per bead — consumer bead.&#xA;- Per-team customization of split rules — premature; ship the default and iterate.&#xA;&#xA;## Anti-goals&#xA;&#xA;- Do **not** rewrite an existing bead in place. Splitting always files children; the parent stays as an epic with `status=blocked-by-children`.&#xA;- Do **not** make the gate silently optional. If disabled via flag or config, emit a one-time warning per worker boot.&#xA;- Do **not** use a model larger than the cheap tier for the gate prompts.&#xA;- Do **not** duplicate routing logic in the gate; it should be a thin caller of the agent SDK with the gate&#39;s two prompts.&#xA;&#xA;## Why P1&#xA;&#xA;The failure mode it prevents — claimed-but-undeliverable beads — corrupts the queue&#39;s claim semantics, wastes operator time, and silently inflates `attempt_count` on epics that will never close. Without this gate, the rest of the routing-cleanup work (`ddx-fdd3ea36` umbrella) ships into a queue with structurally bad inputs.
