    <title>agent: stop synthesizing default routing profile from empty input</title>
    <description>
PROBLEM&#xA;DDx synthesizes a `default` routing profile for any operator who does not pass `--profile`, then sends `Profile=&#34;default&#34;` to fizeau. Fizeau&#39;s `default` profile (`fizeau/internal/modelcatalog/catalog/models.yaml:438-443`) hard-pins `min_power: 7, max_power: 8`. For operators whose model catalog has no power-7-or-8 entries — e.g., local providers (power 5–6) plus frontier cloud (power 9) — `routePowerPolicyForRequest` overlays those bounds and the routing engine rejects every candidate as `below_min_power` or `above_max_power`. Result: zero-config `ddx work` claims a bead, dispatches with empty intent, fizeau returns &#34;no viable routing candidate,&#34; and the agent exits in &lt;1s with `no_evidence_produced`. Power was meant to be a hint, not a gate; synthesizing a constraint the operator never asked for inverts that policy. Repro: in this repo with no flags, `ddx work --once` triggers the dispatch-failure on hooks today and would trigger it on the implementation dispatch too if explicit `--harness/--model` flags weren&#39;t bypassing the bound via fizeau&#39;s exact-pin escape (`routePowerBoundsForRequest`).&#xA;&#xA;ROOT CAUSE WITH FILE:LINE&#xA;- `cli/internal/agent/routing_profile.go:9-15` returns the constant `&#34;default&#34;` whenever the operator-supplied profile is empty.&#xA;- `cli/cmd/work.go:70` registers the `--profile` flag with default value `agent.DefaultRoutingProfile`, so cobra fills in `&#34;default&#34;` even when the operator omits the flag.&#xA;- `cli/cmd/try.go:86` does the same for `ddx try`.&#xA;- `cli/cmd/run.go:137`, `cli/cmd/agent_cmd.go:298`, `cli/cmd/agent_cmd.go:1613`, `cli/cmd/try.go:215`, and `cli/internal/server/workers.go:{303,409,705}` all call `NormalizeRoutingProfile`, inheriting the synthesis.&#xA;- Downstream fizeau gating: `routePowerPolicyForRequest` (`fizeau/service_routing.go:1117-1141`) returns zero bounds only when `req.Profile == &#34;&#34;`; any non-empty profile string causes a catalog lookup that overlays its `min_power`/`max_power` onto the request.&#xA;&#xA;PROPOSED FIX&#xA;- Change `NormalizeRoutingProfile` to return the trimmed input as-is; an empty operator value stays empty. Keep the function (callers want trim+normalize), drop the synthesis branch.&#xA;- Remove the `DefaultRoutingProfile = &#34;default&#34;` constant, or repurpose it strictly for the help-text label rather than a runtime default.
