    <description>
**Reporter**: nexiq (downstream consumer). Observed across 2026-04-22 → 2026-04-26 on master.&#xA;&#xA;## Observed&#xA;&#xA;`ddx agent execute-bead` can land a commit locally (worktree merged into local master), then fail at the `git push origin master` step, and still classify the bead as `execute-bead: success`. The push failure shows up as `landed locally; push failed:` in the event body, but the bead transitions to closed and the loop moves on.&#xA;&#xA;Concrete blast radius: ~969 unpushed commits accumulated on nexiq&#39;s master across multiple days because every bead that landed in that window had a 258 MB turbopack `.sst` file somewhere in its commit history. GitHub rejected each push with the 100 MB-per-file limit error. The execute-loop&#39;s success rate looked normal — operators saw bead = closed, dashboard = working — but origin was untouched.&#xA;&#xA;Recovery required `git filter-branch --index-filter &#39;git rm --cached -r packages/web/.next .next&#39;` across ~1000 commits plus a `.gitignore` hardening change. By that point the only signal that something was wrong was an operator manually running `git rev-list --count origin/master..master`.&#xA;&#xA;## Root cause&#xA;&#xA;The execute-bead orchestrator currently classifies the result of the *merge* step. The push step happens after, and a non-zero exit there is logged in the event body but does not bubble up to the bead&#39;s terminal status. As far as routing/state machines are concerned, the bead is done.&#xA;&#xA;This is the worst-case classification: silent for operators, irreversible for state (bead closes, primary doesn&#39;t re-run, the work that needs to be re-pushed isn&#39;t tracked anywhere).&#xA;&#xA;## Failure modes this masks&#xA;&#xA;- Large blob in commit history (the case we hit) — every subsequent bead inherits the bad ancestor and fails to push.&#xA;- Auth expiration / credential rotation on the origin remote.&#xA;- Branch protection rules added on the remote between the loop&#39;s last successful push and now.&#xA;- HTTP 5xx / network flake against origin.&#xA;- Local divergence from origin (someone else pushed; non-fast-forward refused).&#xA;&#xA;In all five cases the current behavior is: bead closes, loop picks the next bead, the divergence grows monotonically.&#xA;&#xA;## Suggested fix&#xA;&#xA;1. Promote push to a first-class step in execute-bead&#39;s outcome classification.&#xA;   - On non-zero exit from `git push`, emit `execute-bead: push_failed` (or similar terminal that explicitly is NOT success) with the push stderr captured in the event body.&#xA;   - Reopen the bead with `last_status=push_failed` and a retry_after that buys time for an operator to investigate.&#xA;2. Surface push failure on the next claim attempt: when a worker tries to claim a bead in `last_status=push_failed`, refuse the claim and show the operator the prior push stderr — don&#39;t blindly redo the primary work, since the local commit is already there.&#xA;3. (Optional, defense in depth) Have execute-loop maintain a per-repo divergence counter (`git rev-list --count origin/&lt;branch&gt;..&lt;branch&gt;`); if it crosses a threshold (e.g. &gt;5), pause new claims and emit a loop-level warning event.&#xA;&#xA;## Reproduction&#xA;&#xA;1. On a local fork, install a `pre-receive` hook on the remote that rejects all pushes (or use a remote that requires a permission the runner doesn&#39;t have).&#xA;2. Queue any bead and run execute-loop normally.&#xA;3. Observe: bead closes as success despite the push failing. The merge commit lives only on local master.&#xA;4. Confirm: `git rev-list --count origin/master..master` is non-zero, but ddx still routed a new worker to the next bead.&#xA;&#xA;## Files to inspect&#xA;&#xA;- nexiq commit `f2c2ed33` (the .gitignore + .next scrub) is the recovery commit on the downstream side.&#xA;- nexiq&#39;s own follow-up bead (`nexiq-e57b41eb`) lists this as a blocker; I&#39;ll amend it once this ddx bead has an ID.&#xA;&#xA;## Acceptance&#xA;&#xA;- A forced push failure (e.g. pre-receive hook rejecting) results in a bead that does NOT show as closed and shows up in `ddx bead list --status open` with a `last_status=push_failed` field.&#xA;- The operator can see the push stderr in the bead&#39;s events.&#xA;- Subsequent claim attempts on that bead fail loudly until the operator clears the state (retry, force-skip, etc.).
    </description>
