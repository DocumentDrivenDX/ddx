    <title>execute-loop: epics that decline with &#39;decompose me&#39; should mark themselves blocked, not rely on ad-hoc cooldowns</title>
    <description>
Today, when execute-bead returns no_changes with rationale = &#39;this epic is too big, split it into sub-beads&#39;, the loop sets a short execute-loop-retry-after (a few hours) and keeps re-attempting the epic forever. The bead surface gives operators no good knob: &#39;unset retry-after&#39; lets it re-burn, &#39;set retry-after = 30 days&#39; is arbitrary, and there is no signal that the bead is structurally not a single-pass deliverable.&#xA;&#xA;Observed in this session on ddx-fdd3ea36 (the routing point-release epic): two consecutive drain attempts produced identical no_changes recommendations to split into 9 sub-beads. The agent burned ~$1 of claude tokens on the second attempt re-deriving the same conclusion.&#xA;&#xA;What is needed:&#xA;&#xA;1. A first-class outcome from execute-bead: &#39;declined: needs decomposition&#39; (distinct from no_changes/execution_failed). Carries the recommendation in a structured field, not buried in the no-changes-rationale string.&#xA;&#xA;2. When the loop sees &#39;declined: needs decomposition&#39;, it should auto-set the bead to status=blocked (or a new status like needs-decomposition), record the recommended sub-beads as a bead-comment / event, and STOP re-attempting until a human or helix-evolve splits it.&#xA;&#xA;3. Or: detect the loop — if the same bead has produced the same no_changes rationale N times in a row (e.g. 2), auto-mark it blocked with reason &#39;agent declined repeatedly; needs decomposition&#39;.&#xA;&#xA;4. Make &#39;execute-loop-retry-after&#39; inspectable / settable via a first-class CLI flag rather than --set / --unset on a magic key. Document the field. The &#39;unset retry-after&#39; shortcut should NOT exist — operators should clear it via &#39;ddx bead clear-cooldown&#39; or similar.
    </description>
    <acceptance>
1. New ExecuteBeadStatus value or extra field captures &#39;declined: needs decomposition&#39; as a structured outcome (not regex on rationale). 2. Loop policy: a bead that has produced &#39;declined: needs decomposition&#39; is auto-set to status=blocked (or equivalent terminal-for-loop status) and is no longer in the dep-ready execution queue. 3. The recommended sub-beads from the decomposition rationale are captured as a structured event on the bead (kind:decomposition-recommendation), not just inline text. 4. CLI: a first-class command for inspecting and clearing execute-loop cooldowns (e.g. ddx bead cooldown show / clear). The current --set/--unset workflow becomes a power-user fallback. 5. Regression test: a fake executor that returns &#39;declined: needs decomposition&#39; is invoked via the loop; assert (a) bead transitions to blocked after the first declination, (b) no second attempt happens, (c) decomposition recommendation is on the bead as a structured event. 6. Migration: documented in DDx CHANGELOG; existing &#39;execute-loop-retry-after&#39; key continues to work as a power-user override but is deprecated for the decomposition case.
    </acceptance>
