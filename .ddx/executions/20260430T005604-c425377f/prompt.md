    <title>bead-id resolver: malformed IDs when ddx bead create runs from inside an execute-bead worktree (+ flag whether workers should create beads in-band at all)</title>
    <description>
## Two-layer problem&#xA;&#xA;### Layer 1 (mechanical bug)&#xA;&#xA;When a worker invokes `ddx bead create` from inside its execute-bead worktree, the resulting bead IDs are malformed. Reproduced today on this project:&#xA;&#xA;  /home/erik/.local/bin/ddx bead ready  → 11 entries with IDs like&#xA;  `.execute-bead-wt-ddx-44236615-20260429T185004-ffa9adf1-e630aeae`&#xA;&#xA;These were the read-coverage child beads spawned from the ddx-44236615 worker (commit ebeb2464). They have valid descriptions, AC, priorities, and labels — only the ID field is mangled. The pattern `.execute-bead-wt-&lt;parent&gt;-&lt;timestamp&gt;-&lt;short&gt;-&lt;random&gt;` matches the worktree directory name format used by `ddx agent execute-bead`. Most likely failure path: the bead-id resolver consults `os.Getwd()` (or equivalent) to derive a project context prefix, sees the worktree dir, and uses it instead of the real project root + a fresh `ddx-&lt;8hex&gt;` short hash.&#xA;&#xA;Concrete impact:&#xA;- 50+ char IDs are unwieldy in CLI commands (`ddx bead show &lt;id&gt;`, `ddx bead dep add ...`).&#xA;- Cross-project readers and the bead store assume `ddx-&lt;8hex&gt;` prefix matching for tooling (filtering, GraphQL adapters, MCP responses) — these long IDs may sort or filter unexpectedly.&#xA;- The IDs encode ephemeral attempt timestamps, which leaks worker-internal state into a permanent identifier.&#xA;- 12 such beads exist in the tracker as of this filing (grep `.execute-bead-wt-` in `.ddx/beads.jsonl`).&#xA;&#xA;### Layer 2 (design question — separate decision)&#xA;&#xA;Even if the IDs were correctly formed `ddx-&lt;8hex&gt;`, it is not obvious that `ddx bead create` from inside an execute-bead worker is the right pattern at all. Two alternatives the design has not articulated a position on:&#xA;&#xA;**(a) Append to the parent bead.** When the worker discovers sub-tasks, it appends them as structured items in the parent&#39;s description, notes, or a new `kind:discovered-subtask` event. The parent bead becomes the durable record of &#34;what this run found&#34;; future runs (or the operator) can decompose if/when the work warrants its own tracker entries.&#xA;&#xA;**(b) Surface via the execute-bead result.** The agent emits a structured `discovered_subtasks: [...]` field on result.json; the loop / operator decides whether to file new beads. Keeps tracker mutations gated by an explicit decision point.&#xA;&#xA;**(c) Create new beads in-band (current behavior).** The worker calls `ddx bead create` directly. Pro: fastest decomposition. Con: sub-tasks land in the queue without operator review; can flood the queue if the agent over-decomposes; complicates &#34;what was THIS run&#39;s contribution&#34; auditing.&#xA;&#xA;Today&#39;s behavior is (c) but undocumented. The 11 read-coverage children landing as a single agent&#39;s decomposition is exactly the failure mode (c) makes easy: one analysis bead spawned 11 implementation beads in a single run, all P0, all without operator review. The work may be valid — but the pattern needs a position.&#xA;&#xA;## In-scope outcomes&#xA;&#xA;This bead&#39;s job is to surface both layers; the implementation work decomposes into follow-ups.&#xA;&#xA;1. **Concrete fix for layer 1**: bead-id resolver should normalize worktree paths to the real project root before allocating an ID. Test: `ddx bead create` invoked with cwd=`&lt;repo&gt;/.execute-bead-wt-...` produces `ddx-&lt;8hex&gt;`, not `.execute-bead-wt-...-&lt;random&gt;`.&#xA;2. **Mass repair for the 12 existing malformed beads**: either rewrite their IDs to fresh `ddx-&lt;8hex&gt;` (with a redirect or alias map for any external references), or close them as deferred and re-file with proper IDs. Decision deferred until layer 2 is settled.&#xA;3. **Layer 2 decision**: file a separate design bead (or amendment to FEAT-006 / FEAT-010) that picks (a), (b), or (c) and documents the rationale. This bead does NOT pre-decide; it captures the question.&#xA;&#xA;## In-scope files (layer 1 fix)&#xA;&#xA;- `cli/cmd/bead_create.go` (or wherever bead-create resolves project context).&#xA;- `cli/internal/bead/store.go` or `cli/internal/bead/id.go` if ID generation lives there.&#xA;- A unit test that drops cwd into a worktree-shaped dir and asserts the resulting bead id matches `^ddx-[0-9a-f]{8}$`.&#xA;&#xA;(Locate via: `grep -rn &#34;execute-bead-wt\|FindProjectRoot\|projectRoot&#34; cli/internal/bead/ cli/cmd/bead_*.go`.)&#xA;&#xA;## Out of scope&#xA;&#xA;- The 11 read-coverage child beads&#39; actual implementation work — that&#39;s separate beads.&#xA;- Default behavior change for whether workers can/should create beads — that&#39;s the layer-2 design bead.&#xA;- GraphQL or MCP changes to handle long IDs — fix the source instead.&#xA;&#xA;## Acceptance&#xA;&#xA;Layer-1 only (this bead):&#xA;&#xA;1. `cd cli &amp;&amp; go test ./cmd/... ./internal/bead/...` passes after the fix.&#xA;2. New unit test in `cli/internal/bead/` (or `cli/cmd/`) that simulates a bead-create call from `cwd=&lt;project&gt;/.execute-bead-wt-ddx-XXXXX-&lt;timestamp&gt;-&lt;random&gt;` and asserts the new bead&#39;s `id` field matches `^ddx-[0-9a-f]{8}$`. Test fails before the fix, passes after.&#xA;3. `grep -E &#39;&#34;id&#34;:&#34;\.execute-bead-wt-&#39; .ddx/beads.jsonl` returns either 0 (mass repair) or 12 with a recorded decision in this bead&#39;s notes that the existing 12 are deferred to a follow-up bead.&#xA;4. A separate bead is filed (or referenced existing one updated) that captures the layer-2 design question (a/b/c). Reference its ID in this bead&#39;s events.&#xA;&#xA;## Why one bead, not an epic&#xA;&#xA;Layer 1 is a tight 1-3 file fix. Layer 2 is a design discussion that should be its own artifact, not a sibling task. Splitting them keeps each bounded.
