HTTP REST endpoints properly enforce project boundaries via scoped()/legacy() middleware (cli/internal/server/server.go:712-719) and workingDirForRequest() (cli/internal/server/server.go:651-656). The /graphql route is registered as `trusted(&#34;POST /graphql&#34;, ...)` not `scoped()`, so it cannot accept a {project} path parameter. The HIGH-severity exposure is at cli/internal/server/graphql/resolver_documents.go:26 — DocumentByPath always reads from r.WorkingDir.&#xA;&#xA;LAYER 1 SCOPE&#xA;Add a parallel route /api/projects/{project}/graphql:&#xA;- Registered via scoped() middleware&#xA;- Resolves the {project} path param via existing project context infrastructure&#xA;- Reconstructs the GraphQL resolver with the request&#39;s project&#39;s WorkingDir per request (or threads it via context to the existing resolver)&#xA;- Existing /graphql endpoint behavior unchanged for backward compat&#xA;&#xA;This is the immediate security fix. Frontend routes /nodes/[nodeId]/projects/[projectId]/... can switch their GraphQL client to point at the scoped endpoint when projectId is present.&#xA;&#xA;NOT IN LAYER 1 (deferred to LAYERS 2 + 3 — separate beads)&#xA;- LAYER 2: refactor resolver to take WorkingDir from request context across ALL resolvers (not just per-route reconstruction). Allows the existing /graphql endpoint to safely serve multiple projects.&#xA;- LAYER 3: implement Bead/Execution resolution in Node(id) Relay lookup with project-aware lookups (resolver_node.go:86-101 currently returns nil for bead IDs).&#xA;&#xA;WHY DECOMPOSE&#xA;- LAYER 1 ships the security fix in one bead worth of work&#xA;- LAYER 2 is a larger refactor that doesn&#39;t need to block the security fix&#xA;- LAYER 3 is completeness; not security-critical&#xA;- All three could land independently if needed (LAYER 2/3 don&#39;t depend on each other)
