PROBLEM&#xA;Orphaned helper processes (e.g. a stray `find`) whose cwd is a SUBDIRECTORY of a non-`.execute-bead-wt-*` scratch/execution root under the configured temp root (not the root itself) are never reaped by ExecutionCleanupManager.CleanupAttemptProcesses, so they survive indefinitely (observed: orphan &#39;find&#39; processes 23+ days old, ddx-a9218e18; design: docs/helix/02-design/technical-designs/TD-040-leftover-worktree-process-reaping.md, &#34;Change 1&#34;).&#xA;&#xA;ROOT CAUSE&#xA;executionCleanupAttemptProcessFromWorkerStatus (cli/internal/agent/execution_cleanup_process.go:415-437) resolves Worktree two ways. The primary path (workerstatus.InferBead + executionCleanupAttemptWorktreeRoot, execution_cleanup_process.go:387-409) correctly truncates any `.execute-bead-wt-*`-containing path to its root, no matter how deep the cwd is nested. The fallback path (execution_cleanup_process.go:420-426), used when cwd has NO `.execute-bead-wt-*` segment but IS inside tempRoot, instead does `worktree = filepath.Clean(trimmedCwd)` -- the raw, untruncated cwd. Downstream, classifyExecutionCleanupAttemptProcessGroup (execution_cleanup_process.go:201-332) requires an EXACT match: either ReadExecutionCleanupMetadata(proc.Worktree) finds a file literally at `&lt;Worktree&gt;/&lt;metadata-filename&gt;` (execution_cleanup.go:1135), or matchingRunStateForMeta (execution_cleanup.go:877-900) does filepath.Clean string-equality against RunState.WorktreePath (execution_cleanup.go:895). When Worktree is a nested subdirectory instead of the owning root, both lookups miss and the process is permanently classified `preserved_uncertain_attempt_process` (execution_cleanup_process.go:266-272) -- never reaped.&#xA;&#xA;PROPOSED FIX&#xA;In the fallback branch of executionCleanupAttemptProcessFromWorkerStatus (execution_cleanup_process.go:420-426), truncate trimmedCwd to the first path component directly under tempRoot (i.e. `filepath.Join(tempRoot, firstComponentAfterTempRoot)`) before assigning Worktree, instead of using the raw cwd. This mirrors executionCleanupAttemptWorktreeRoot&#39;s truncate-at-next-separator rule but anchors at tempRoot instead of the `.execute-bead-wt-*` prefix.&#xA;&#xA;NON-SCOPE&#xA;- No change to the primary `.execute-bead-wt-*` attribution path (execution_cleanup_process.go:415-419) -- it already truncates correctly.&#xA;- No open-fd attribution fallback -- that is a separate child bead (see TD-040 &#34;Change 2&#34;).&#xA;- No change to orphan_harness_reaper.go&#39;s cmdline-allowlist path.&#xA;- No change to kill semantics (still process-group SIGKILL via the existing KillGroup).
