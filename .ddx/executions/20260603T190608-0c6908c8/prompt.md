    <title>execute-bead: harness child orphans to PID 1 when ddx work parent dies abnormally (SIGKILL/crash) — no Pdeathsig, no startup reaper</title>
    <description>
PROBLEM&#xA;When a `ddx work` orchestrator dies abnormally (SIGKILL, crash, OOM, host reboot of the supervising process) while a bead attempt is in flight, the spawned harness child (e.g. `claude --print -p --verbose --output-format stream-json ... &lt;execute-bead&gt;` or `codex exec --json ...`) is NOT reaped. It reparents to PID 1 and keeps running indefinitely. Observed 2026-05-27 on the dev host: a `claude --print` execute-bead child orphaned with PPID 1 for ~9h53m after its `ddx work` parent had gone — burning a harness session and (per the sibling fizeau leak) anchoring leaked subprocess trees. This is the orchestrator-side mirror of fizeau-50b6f055 (harness-runner-side leak).&#xA;&#xA;ROOT CAUSE (file:line)&#xA;- cli/internal/agent/executor.go:156 — OSExecutor.ExecuteInDir spawns the harness via exec.Command(binary, args...) then cmdSetProcessGroup(cmd).&#xA;- cli/internal/agent/executor_unix.go:13-15 — cmdSetProcessGroup sets ONLY SysProcAttr{Setpgid: true}. There is NO Pdeathsig. So the kernel does nothing automatically when the parent dies.&#xA;- cli/internal/agent/executor_unix.go:20-25 — cmdKillProcessGroup(cmd) SIGKILLs the negative pgid, but it is only invoked from the executor&#39;s graceful paths (stopProcess at executor.go:190-194, triggered by idle/wall-clock timeout, auth/rate-limit pattern, or ctx cancellation at executor.go:231-255).&#xA;- cli/cmd/root.go:49-54 — the worker context is a signal.NotifyContext on os.Interrupt + SIGTERM only. Graceful shutdown works; SIGKILL and crashes cannot be caught, so the defer/stopProcess chain never runs and the child survives.&#xA;- Net: child harness lifetime is governed entirely by parent-run cleanup. If the parent does not get to run cleanup, the child orphans. There is no kernel-enforced parent-death signal and no startup reaper that adopts/kills harness children whose parent PID is gone.&#xA;&#xA;SECONDARY VECTOR&#xA;- cli/internal/agent/attempt_backend.go:293 — the docker-clone backend spawns `exec.CommandContext(ctx, &#34;docker&#34;, ...)` WITHOUT cmdSetProcessGroup, so even graceful cleanup may not reach the nested harness inside the container. (worktree/local-clone backends at attempt_backend.go:120-122 / :182-184 share the executor.go:156 path and do get Setpgid.)&#xA;&#xA;PROPOSED FIX&#xA;1. Primary (Linux): set SysProcAttr.Pdeathsig = SIGKILL (or SIGTERM) in cmdSetProcessGroup so the kernel signals the harness child when the parent thread dies, regardless of how the parent died. Note the well-known caveat: Pdeathsig keys off the spawning OS *thread*, so pin/guard against Go runtime thread migration (runtime.LockOSThread around Start, or re-check parent PID in the child path). Document the macOS/Windows gap (Pdeathsig is Linux-only) and the chosen fallback there.&#xA;2. Backstop: on `ddx work` startup, reap orphaned harness children from prior dead workers — e.g. when registering/claiming, scan for execute-bead harness processes whose parent is PID 1 (or whose recorded owner PID is gone) and SIGKILL their process groups, gated to this project/workspace to avoid nuking unrelated work. Ties to the existing lease-release / operator-attention machinery (cross-ref ddx-8f2e0ebf).&#xA;3. docker-clone backend: apply cmdSetProcessGroup to the docker invocation at attempt_backend.go:293 (and/or run with --rm + a container-stop on cleanup) so the container tree is reaped too.&#xA;&#xA;NON-SCOPE&#xA;- Graceful SIGINT/SIGTERM shutdown already works (executor.go:231-255); do not regress it.&#xA;- The harness-runner-internal leak in fizeau is covered by fizeau-50b6f055; this bead is the ddx orchestrator -&gt; harness-child boundary.&#xA;&#xA;CROSS-REF: fizeau-50b6f055 (fizeau harness-runner process-group cleanup), ddx-8f2e0ebf (lease release / operator-attention).
