    <title>agent executor: enforce Linux parent-death cleanup for execute-bead harness children</title>
    <description>
PROBLEM&#xA;When a `ddx work` worker dies abnormally while an execute-bead harness is running, the direct harness child can reparent to PID 1 and continue burning a harness session. This was observed on 2026-05-27 with a `claude --print` execute-bead child orphaned under PPID 1 for about 9h53m.&#xA;&#xA;ROOT CAUSE&#xA;- cli/internal/agent/executor.go:156 spawns the harness with exec.Command(binary, args...) and then calls cmdSetProcessGroup(cmd).&#xA;- cli/internal/agent/executor_unix.go:13-15 sets only SysProcAttr{Setpgid: true}; it does not configure a parent-death signal.&#xA;- cli/cmd/root.go:49-54 handles os.Interrupt and SIGTERM only. SIGKILL, crashes, OOM, and parent process death cannot run the normal defer/stopProcess cleanup.&#xA;- cli/internal/agent/executor.go:190-194 and cli/internal/agent/executor.go:231-255 only kill the process group when graceful executor paths run.&#xA;&#xA;PROPOSED FIX&#xA;- On Linux, extend cmdSetProcessGroup in cli/internal/agent/executor_unix.go so execute-bead harness children receive a kernel parent-death signal, preferably SIGKILL unless implementation chooses SIGTERM with a defensible reason.&#xA;- Handle the Go runtime thread-migration caveat for SysProcAttr.Pdeathsig, for example by locking the OS thread around process start or by an equivalent parent-PID guard that makes the death signal reliable.&#xA;- Add a non-Linux fallback/rationale in platform-specific code or docs/tests so macOS and Windows behavior is explicit.&#xA;- Preserve the existing Setpgid behavior because graceful shutdown still relies on process-group termination.&#xA;&#xA;NON-SCOPE&#xA;- Do not implement startup orphan reaping in this child; that is covered by the startup reaper child bead and cross-ref ddx-8f2e0ebf.&#xA;- Do not change docker-clone backend cleanup; that is covered by the docker-clone child bead.&#xA;- Graceful SIGINT/SIGTERM shutdown already works through executor.go:231-255; do not regress it.&#xA;- The harness-runner-internal leak in fizeau is covered by fizeau-50b6f055; this child remains at the ddx orchestrator -&gt; harness-child boundary.&#xA;&#xA;GOVERNING ARTIFACTS / CROSS-REFS&#xA;- docs/helix/06-iterate/bead-authoring-template.md&#xA;- docs/helix/06-iterate/reliability-principles.md&#xA;- fizeau-50b6f055&#xA;- ddx-8f2e0ebf
