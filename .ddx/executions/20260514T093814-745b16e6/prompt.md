PROBLEM&#xA;`ddx work` and `ddx try` execute beads inline without writing any worker-status record to `.ddx/workers/`. The server&#39;s WorkerManager writes those records only for server-started workers. For inline workers, operators cannot see worker_id, current bead, attempt_id, current phase, or last_activity_at — the only visible signal is `extra.claimed-at` on the bead, which never refreshes. This is the core information gap for &#34;is this worker alive and what is it doing?&#34; during a long-running attempt.&#xA;&#xA;ROOT CAUSE WITH FILE:LINE&#xA;- `cli/internal/agent/work/heartbeat.go:13-35` periodically calls `store.TouchClaimHeartbeat` but has no hook for writing operator-facing worker status.&#xA;- `cli/cmd/execute_loop_shared.go:154-186` constructs the inline `ExecuteBeadWorker` and `workerprobe.Probe` but never generates a local worker_id or writes a `.ddx/workers/&lt;id&gt;/status.json` sidecar.&#xA;- `cli/internal/agent/execute_bead_loop.go:1585-1620` tracks `provAttemptID` and `phaseSeq` per attempt but does not propagate them to any persistent status file.&#xA;- `cli/internal/server/workers.go:1688-1713` defines the WorkerRecord schema used by the server; inline workers should use a compatible subset (worker_id, current_bead, current_attempt, started_at, last_activity_at, pid) so existing consumers (`workerDirPID` at `cli/internal/agent/execution_cleanup.go:1250-1262`, `pruneWorkerDirs` at line 1267) keep working.&#xA;&#xA;PROPOSED FIX&#xA;Introduce a `LivenessReporter` in `cli/internal/agent/work/` (or a thin helper package under `cli/internal/agent/workerstatus/`) with:&#xA;- `Update(now time.Time, info LivenessInfo)` — atomic write to `.ddx/workers/&lt;worker-id&gt;/status.json` (write to tmp + rename); concurrent-safe; idempotent for repeated identical state&#xA;- Fields persisted: worker_id, project_root, current_bead, attempt_id, phase, started_at, last_activity_at, harness/model/profile summary, pid&#xA;Wire a tick callback into `work.WithHeartbeat` so each tick calls `Update`.
