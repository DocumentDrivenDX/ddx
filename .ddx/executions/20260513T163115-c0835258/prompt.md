    <title>routing: send MinPower only for review and fallback escalation</title>
    <description>
PROBLEM&#xA;DDx currently duplicates Fizeau policy intent with numeric power on initial zero-config implementation dispatch. In the current code path, DDx selects the `default` Fizeau policy for ordinary implementation work and then also sets `MinPower` to that policy floor. That makes the request shape `Policy=default, MinPower=7` instead of letting Fizeau fully interpret the selected policy.&#xA;&#xA;ROOT CAUSE WITH FILE:LINE&#xA;`cli/cmd/execute_loop_shared.go:396-401` selects an implementation profile and raises `initialMinPower` to `selection.MinPower` on the first attempt. `cli/cmd/execute_loop_shared.go:426-442` then passes `requestedMinPower` into `singleTierAttempt`, which writes it into `config.CLIOverrides.MinPower` at `cli/cmd/execute_loop_shared.go:289-294`. `cli/internal/agent/service_run.go:241-258` sends both `Policy` and `MinPower` to Fizeau. The same first-attempt pattern exists for `ddx try` at `cli/cmd/try.go:293-322`. Current command tests assert this duplicated shape in `cli/cmd/work_test.go` and `cli/cmd/try_test.go`, so the tests encode the wrong contract.&#xA;&#xA;DESIRED ROUTING CONTRACT&#xA;For initial zero-config implementation work, DDx should select an opaque Fizeau policy by metadata and send only that policy. Example: ordinary work selects `Policy=default`, `MinPower=0`. Fizeau owns the model choice and may choose a lower-power model inside that policy if the policy permits it.&#xA;&#xA;For implementation fallback escalation, DDx should keep policy intent and add a numeric floor only after evidence of failure. Example: DDx selects `Policy=default`; Fizeau chooses a power-5 model; the attempt fails; DDx retries with `Policy=default, MinPower=6`. If repeated failures or ladder exhaustion require moving policy bands, DDx may reselect the appropriate policy by metadata, but numeric floors are evidence-driven retry constraints, not initial policy duplication.&#xA;&#xA;For review routing, DDx may use `MinPower` immediately when the reviewer must be stronger than the implementation route or when review pairing evidence requires it.&#xA;&#xA;PROPOSED FIX&#xA;Change zero-config implementation routing so first attempts do not copy the selected policy floor into `MinPower`. Preserve explicit operator `--min-power`, bead numeric tier hints, reviewer escalation, and retry escalation. On retry, derive the next floor from the actual route power when available; otherwise use the selected policy floor/ladder only as an escalation basis, not as an initial dispatch floor.&#xA;&#xA;NON-SCOPE&#xA;Do not hard-code Fizeau policy names. Do not add DDx-side provider/model selection. Do not change Fizeau policy definitions. Do not remove MinPower support for explicit user flags, review escalation, or implementation retry escalation.
    </description>
