    <description>
PROBLEM: The DDx main-git lock (`.ddx/.git-tracker.lock`,&#xA;`cli/internal/agent/tracker_lock.go`) is held across operations that do&#xA;not require atomic serialization. The two worst offenders are&#xA;pre-dispatch (`execute_bead.go:1493-1524`) and Land&#xA;(`execute_bead_land.go:1210-1216`). On a niflheim-sized repo under disk&#xA;contention, the held region can exceed the 30s&#xA;`MaxElapsed` lock-wait budget, causing sibling workers to time out at&#xA;`withTrackerLockPolicy` line 205 (&#34;max elapsed&#34;) and the loop to&#xA;misclassify the timeout (sibling bead ddx-2189a07f).&#xA;&#xA;ROOT CAUSE: Two regions hold the lock around work that is sequencing,&#xA;not critical section:&#xA;&#xA;  Pre-dispatch at `execute_bead.go:1493-1524`:&#xA;  - commitTrackerLocked(projectRoot)             — needs lock (writes .ddx/beads.jsonl + commits)&#xA;  - checkpointPreDispatchDirt(projectRoot, ...)   — needs lock (commits on branch)&#xA;  - resolveBase(gitOps, projectRoot, ...)         — read-only ref resolve; reads must observe the commits above&#xA;  - attemptBackend.Prepare(...)                   — creates worktree via `git worktree add --detach &lt;wt&gt; &lt;baseRev&gt;`; writes per-worktree state under .git/worktrees/&lt;id&gt;/ — NO shared-state conflict with sibling workers&#39; Prepare()&#xA;&#xA;  Land at `execute_bead_land.go:1208-1216` calls `landLocked` which&#xA;  performs (per `:1218-1259` and beyond):&#xA;  - resolve target branch, target ref, current tip                 — read-only&#xA;  - countContributingCommits / preserveIfLargeDeletion              — read-only / preservation stash&#xA;  - ensureLandingWorktreeReady                                       — sets up a temp worktree&#xA;  - the actual fast-forward / merge commit on target branch         — needs lock (ref update)&#xA;  - dirty-path sync to operator checkout                             — outside merge correctness&#xA;&#xA;  In both cases the ONLY operations that race with sibling workers are&#xA;  the ref/tracker writes. Reads can be ordered by re-resolving after&#xA;  reacquiring the lock if the write decision depends on them.&#xA;&#xA;The conservative lock placement appears to be defensive rather than&#xA;correctness-required. There is no comment at either call site explaining&#xA;why worktree creation or merge prep must be inside the lock, only the&#xA;single-line claim at `execute_bead.go:1489-1490` that the lock is &#34;held&#xA;only across the brief pre-dispatch sequence (~1-2s)&#34; — which the&#xA;observed 30s timeout disproves.&#xA;&#xA;PROPOSED FIX:&#xA;&#xA;A) Pre-dispatch (`execute_bead.go:1493`):&#xA;   1. Inside the lock, re-resolve `currentTip` and verify the prep-phase&#xA;      decision is still valid (target branch hasn&#39;t moved). If it has&#xA;      moved, release and retry the prep phase. This is the classic&#xA;      optimistic-locking shape.&#xA;   3.
