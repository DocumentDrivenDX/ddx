    <title>server: warn about orphaned home .ddx server and tsnet state</title>
    <description>
PROBLEM: After `ddxroot.Path($HOME)` stops returning `~/.ddx/`, existing incorrectly-created home state such as `~/.ddx/server/` and `~/.ddx/tsnet/` becomes orphaned. Operators need an explicit startup signal so stale TLS or tsnet state does not keep confusing client/server trust debugging.&#xA;&#xA;ROOT CAUSE WITH FILE:LINE: `cli/internal/ddxroot/ddxroot.go:Path()` previously allowed `~/.ddx/` to act as a project state root. High-risk server consumers include TLS cert/key state at `cli/internal/server/server.go:326` and tsnet state at `cli/internal/server/server.go:503`. Once the resolver falls through to XDG, those old home directories are no longer read by the server path helpers.&#xA;&#xA;PROPOSED FIX: Add startup detection in the server path that checks for legacy `JoinHome(&#34;server&#34;)` and `JoinHome(&#34;tsnet&#34;)` directories when the active project state root resolves elsewhere. Emit a clear warning naming the orphan source and the active XDG/project state destination. Keep the behavior warning-only unless the implementation already has an established safe migration helper.&#xA;&#xA;NON-SCOPE: Do not rename `~/.ddx/` to `~/.config/ddx/`. Do not address multi-project federation collisions. Do not change TLS-renew scheduling or rotation logic. Do not silently delete legacy home-state directories.
