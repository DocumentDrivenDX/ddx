PROBLEM&#xA;Operators need to say &#39;project X should have N queue-drain workers&#39; and have DDx keep that true across server ticks and restarts. Today the server can start a worker once, but the desired count and restart policy are not durable.&#xA;&#xA;ROOT CAUSE WITH FILE:LINE&#xA;cli/internal/server/workers.go:192 defines WorkerManager as an in-memory goroutine manager with workers map state. cli/internal/server/workers.go:361 StartExecuteLoop writes per-worker spec/status, but there is no project-level desired-state file such as .ddx/workers/desired.json. cli/internal/server/workers.go:1971 ReconcileStaleWorkers handles stale records, not desired count, restart policy, or backoff.&#xA;&#xA;PROPOSED FIX&#xA;Add a small desired-state model and persistence layer under .ddx/workers/desired.json, then add a reconcile component around WorkerManager. The reconciler should load desired_count, default worker spec, restart enabled/backoff/max restarts, compare desired server-managed workers to actual server-owned workers, start missing workers, stop excess workers, mark stale running records stopped, and pause restart on dirty-root/operator-attention blockers.&#xA;&#xA;NON-SCOPE&#xA;Do not implement process-group child cleanup here beyond using existing Stop.
