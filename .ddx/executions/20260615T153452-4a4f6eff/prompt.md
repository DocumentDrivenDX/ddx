    <title>server: terminate managed worker process trees without leaks</title>
    <description>
PROBLEM&#xA;A server-managed worker must be fully cleanable. Stop, watchdog reap, restart, and server shutdown must not leave ddx work, claude, codex, shell, or sleep descendants behind.&#xA;&#xA;ROOT CAUSE WITH FILE:LINE&#xA;cli/internal/server/workers.go:192 says WorkerManager manages in-process work workers as goroutines. cli/internal/server/workers.go:438 creates a cancelable context, but no OS process boundary. cli/internal/server/workers.go:1234 reads WorkerRecord.PID and :1282-1286 terminates a process group only when pid &gt; 0. cli/internal/server/workers.go:1453-1457 has the same limitation in watchdog reap. Because server-owned workers are often goroutines with PID zero, the server cannot guarantee child harness cleanup.&#xA;&#xA;PROPOSED FIX&#xA;Introduce a server-managed process boundary. Preferred implementation: spawn a child ddx work --server-managed &lt;worker-id&gt; in its own process group and record PID/PGID in WorkerRecord. If a transitional in-process path remains for tests, every harness child process group must be registered on the worker handle and killed through the same cleanup helper. Stop/reap/shutdown should send SIGTERM, wait WatchdogKillGrace, SIGKILL remaining descendants, release bead claims, persist terminal status, and be idempotent.&#xA;&#xA;NON-SCOPE&#xA;Do not kill external reported workers or interactive Claude/Codex sessions. Do not change bead claim semantics except releasing claims for the stopped/reaped managed worker. Do not implement UI controls.
