    <title>picker: a transient tracker-lock-timeout skip of a higher-priority bead must not fall through to a lower one (queue-order violation)</title>
    <description>
PROBLEM: ddx workers do not strictly respect queue/priority order — they skip higher-priority beads and claim lower/&#34;easier&#34; ones (operator-observed: &#34;picks up all the easy beads, not the next&#34;). Evidence: 485 &#34;tracker lock timeout (skipping ddx-XXX)&#34; entries across the worker logs; the picker already emits a picker.priority_skip diagnostic (cli/internal/agent/execute_bead_loop.go:1623-1632) precisely because it claims a lower bead while higher ones were skipped.&#xA;&#xA;ROOT CAUSE: nextCandidate (cli/internal/agent/execute_bead_loop.go:4100) iterates ready beads in priority/queue order and selects the FIRST CLAIMABLE candidate, recording the skipped higher beads + reasons. Many of those skips are TRANSIENT tracker-lock-timeouts (lockTimeoutError, cli/internal/agent/tracker_lock.go:297) — the pre-claim readiness check could not acquire the shared tracker lock because a peer worker holds it during a long operation (the multi-minute land_post_land_finalize holds, see ddx-177a20c4 / ddx-5e8dac57). A transient &#34;couldn&#39;t get the lock this instant&#34; is treated the SAME as a genuine skip (wedged / label-filtered / unready / lost-race), so the picker FALLS THROUGH to a lower-priority claimable bead. preClaimIdleSkip (execute_bead_loop.go:3748) only routes to the sleep-and-retry-from-top idle path when ALL skips are idle-class; a transient lock-timeout on a higher bead while a lower bead is claimable does NOT trigger it, so the lower bead is claimed → strict order violated. isTransientGitContention (execute_bead_loop.go:1119) already classifies TrackerLockTimeoutErr as transient for the COMMIT/claim retry path, but the PICKER&#39;s skip classification does not use it to block fall-through.&#xA;&#xA;PROPOSED FIX: In nextCandidate (or its caller), when a HIGHER-priority bead is skipped for a TRANSIENT reason (tracker-lock-timeout / the isTransientGitContention sentinel — the bead IS claimable, just lock-contended this instant), do NOT select a lower-priority bead beneath it. Return &#34;no claimable candidate yet (transient)&#34; so the loop takes the sleep+backoff+re-scan-from-top path (mirroring preClaimIdleSkip), preserving strict priority/queue order. Only fall through past a higher bead for GENUINE skips: consecutive-wedge, label-filtered, lost-race-to-another-worker (winner holds a fresh lease), or genuinely-unready. Add the transient-lock-timeout skip-reason to preClaimIdleSkip&#39;s idle-class set (or an equivalent transient gate).&#xA;&#xA;NON-SCOPE: the lock-hold reduction (ddx-177a20c4/5e8dac57 — reduces how OFTEN this fires); the claim-exclusivity/heartbeat fixes (ddx-d333ad8a/ba93de9e); genuine head-of-line skips (wedge/filter/unready) which SHOULD still fall through to keep the queue draining.
