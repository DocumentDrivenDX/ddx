PROBLEM&#xA;A malformed `.ddx/.git-tracker.lock` path can stop `ddx work` before it claims any bead. The observed incident was a zero-byte regular file at `.ddx/.git-tracker.lock`; `ddx work` then printed repeated lines like `pre-claim hook: tracker lock timeout (max elapsed, owner pid: unknown) (skipping ddx-...)` for every ready candidate.&#xA;&#xA;ROOT CAUSE WITH FILE:LINE&#xA;- cli/internal/agent/tracker_lock.go:130 calls `os.Mkdir(lockDir, 0o755)` and treats every failure as ordinary lock contention.&#xA;- cli/internal/agent/tracker_lock.go:139-149 only calls `breakStaleTrackerLock(lockDir)` and then sleeps/retries. That stale path only understands a lock directory with `pid` and `acquired_at` sidecars, so a regular file at the lock path is never classified or removed.&#xA;- cli/internal/agent/tracker_lock.go:156-161 reports `owner pid: unknown` by trying to read `lockDir/pid`, which is impossible when `lockDir` is a regular file.&#xA;- docs/helix/02-design/technical-designs/TD-031-bead-state-machine.md:499-522 now specifies the required filesystem-shape contract: `.ddx/.git-tracker.lock` is a lock directory;
