    <description>
PROBLEM&#xA;On provider_connectivity failure, applyProviderConnectivityRouteExclusion calls nextFloorFn(actualPower). When the configured power ladder has no step above actualPower (common when the local config exposes only cheap-tier providers), nextFloorFn returns powerladder.ErrLadderExhausted. The current code silently swallows that error and does NOT set b.Extra[TriagePowerHintKey]. As a result the next claim sees no hint, the routing intent re-derives &#34;cheap&#34;, and the same dead (provider, model) is selected again. The failed-route list is appended to but never consulted by Fizeau, so it has no effect on the next decision. A drain in this state attempts each bead once, times out for ~2m45s in Fizeau retries, gets the same provider_connectivity outcome, and bottoms out at 100% failures with no escalation ever taking effect. Reproduces exactly when local LM Studio hosts (bragi/grendel) are unreachable: every bead routes to power=5 cheap models, fails, retry-after expires 15min later, re-routes to the same model, fails again.&#xA;&#xA;ROOT CAUSE&#xA;cli/internal/agent/execute_bead_loop.go:3862-3866 in applyProviderConnectivityRouteExclusion:&#xA;    nextFloor, err := nextFloorFn(report.ActualPower)&#xA;    if err != nil {&#xA;        return&#xA;    }&#xA;    b.Extra[TriagePowerHintKey] = nextFloor&#xA;&#xA;The early return swallows ErrLadderExhausted. The same silent-swallow pattern also lives in applyNoChangesSmartRetry (execute_bead_loop.go:3454) and applyRepairCycleExhaustedEscalation (execute_bead_loop.go:3497). The bead-event log gets a route-failure entry but no operator-visible signal that escalation aborted.&#xA;&#xA;PROPOSED FIX&#xA;- When nextFloorFn returns ErrLadderExhausted (or any error), emit a kind=execution-escalation-aborted event with a summary naming the failed (provider, model, actualPower) and the reason &#34;ladder exhausted at power N&#34;.&#xA;- Set TriagePowerHintKey to actualPower+1 anyway (or to a sentinel &gt;=actualPower+1) so the next route decision at least knows &#34;anything but the failed tier&#34; — this is a no-op for routers that ignore the hint, but it gives clear signal to Fizeau-side exclusion honoring (separate bead).&#xA;- After 2 consecutive provider_connectivity failures on the SAME (provider, model), promote the bead&#39;s outcome to operator_required with a clear message (&#34;provider X / model Y unreachable on 2+ attempts; check ddx config or provider health&#34;) so the queue does not silently re-loop forever.&#xA;- Apply the same change to applyNoChangesSmartRetry and applyRepairCycleExhaustedEscalation so all three escalation paths surface ladder exhaustion uniformly.&#xA;&#xA;NON-SCOPE&#xA;- Do not change Fizeau-side routing exclusion (separate bead in upstream repo).&#xA;- Do not introduce a new bead status; reuse open/blocked/operator_required.&#xA;- Do not auto-edit operator config; just signal clearly.&#xA;&#xA;DEPS&#xA;None.
