    <title>server: singleton-per-machine guard + CLI validates pid-alive before trusting server.addr</title>
    <description>
Two related bugs causing &#39;ddx server workers list&#39; (and other server-talking CLI commands) to fail with &#39;connection refused dial tcp 127.0.0.1:&lt;stale-port&gt;&#39; even when a live server is running.&#xA;&#xA;## Diagnosis&#xA;&#xA;**writeAddrFile already exists** at cli/internal/server/server.go:287-312 and is called from line 273 on every startup. So the file IS written. The real bugs are:&#xA;&#xA;**Bug A: No singleton-server guard.** Multiple &#39;ddx server&#39; instances can start on the same machine concurrently. Each one calls writeAddrFile and overwrites the previous record. When one dies (e.g., a manual launch test), it leaves the address-file pointing to itself even though it&#39;s no longer running. systemd unit (Type=simple) prevents systemd-spawned duplicates of the same unit name but does NOT stop a manual &#39;ddx server&#39; from racing.&#xA;&#xA;Observed today: live systemd server PID 3442304 listening on 7743 (started ~14:27), but address file shows PID 3482077 / port 4174 (started ~15:01, died). Some manual or test launch overwrote the file then died.&#xA;&#xA;**Bug B: CLI doesn&#39;t validate pid-alive before trusting the address file.** ReadServerAddr at cli/internal/server/server.go:328+ returns whatever&#39;s in the file. CLI commands then dial the recorded URL. If the recorded pid is dead, dial fails. No fallback to the systemd default (127.0.0.1:7743), no warning.&#xA;&#xA;## Fix&#xA;&#xA;Two halves, same surface (server bootstrap + CLI client read path):&#xA;&#xA;**Server-side (singleton guard):**&#xA;- On startup, server acquires flock on ~/.local/share/ddx/server.lock&#xA;- If lock is held by an alive process: refuse to start with clear error (&#39;ddx server already running, pid=&lt;N&gt;, addr=&lt;URL&gt;; only one ddx server per machine&#39;). Exit non-zero.&#xA;- If lock is acquired (or stale — held by dead pid): proceed; release-on-exit handler clears the lock.&#xA;- writeAddrFile happens AFTER the lock acquires — so the address file always corresponds to the lock-holder.&#xA;&#xA;**Client-side (defense in depth):**&#xA;- ReadServerAddr (or its callers) validates the recorded pid is alive (kill -0 equivalent on Unix, OpenProcess on Windows)&#xA;- If pid is dead: emit warning (&#39;server.addr points to a dead pid &lt;N&gt;; falling back to default 127.0.0.1:7743&#39;) and use the default&#xA;- Optionally clear the stale file (best-effort; don&#39;t fail if unwritable)&#xA;&#xA;## Out of scope&#xA;&#xA;- Multi-server-per-machine support (the singleton guard is intentional — one server per machine is the design per FEAT-020)&#xA;- Federation: hub/spoke topology (Story 14) is multi-node, NOT multi-server-per-node
    </description>
    <acceptance>
1. Server acquires flock on ~/.local/share/ddx/server.lock at startup. 2. Second &#39;ddx server&#39; launch refuses with clear error message naming the existing pid + addr; exits non-zero. 3. Lock release on graceful shutdown (SIGTERM); stale-lock detection via dead-pid check on next startup. 4. writeAddrFile only runs after lock acquisition (file is consistent with lock holder). 5. ReadServerAddr (or its callers) validates recorded pid is alive before connecting. 6. If pid is dead, CLI emits a warning and falls back to default 127.0.0.1:7743. 7. New tests in cli/internal/server/: TestServer_SingletonGuardRefusesSecondLaunch + TestServer_StaleAddressFallbackWithWarning. 8. Manual verification: kill the live server; launch a new one; address-file matches the new server. Launch a second &#39;ddx server&#39; while first is running; second refuses. &#39;ddx server workers list&#39; succeeds against the live server after this fix.
    </acceptance>
