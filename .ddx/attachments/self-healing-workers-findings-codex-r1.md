### Findings

| Severity | Area | Finding |
|---|---|---|
| BLOCKING | Slot identity | `project_root + profile + desired_index` is under-specified. It does not define canonicalization for symlinks, moved repos, case-sensitive vs case-insensitive paths, deleted projects, renamed profiles, or config changes that shrink/expand desired worker count. Different agents could create duplicate slots or orphan old slots. |
| BLOCKING | State ownership | The plan does not define the durable store, schema migration, locking model, or single-writer rule for worker slots and runtime JSONL. This is especially risky because the target already says “Runtime JSONL telemetry can conflict during concurrent worker sync.” |
| BLOCKING | Unsafe automation | The meta-scan “may remove dead-PID locks, clear stale run-state, unclaim stale claims, park/block through lifecycle rules, restart missing desired workers, and file code-defect beads,” but it does not define proof thresholds. Dead PID alone is not enough across PID reuse, stale heartbeat clock skew, host reboot, detached sessions, or workers running outside the server. |
| BLOCKING | Lifecycle invariant | `claim -> terminal outcome -> close/unclaim/block/propose` is not complete enough to implement. It does not define which terminal reasons map to `unclaim`, `block`, or `propose`, nor who owns the transition when the provider exits, readiness decode fails, a child never starts, or a process dies mid-claim. |
| BLOCKING | Review gate | The Cayce preserved-needs-review incident is named, but the fix does not define the invariant preventing a `preserved-needs-review` bead with a large-deletion gate from remaining worker-eligible. “Park/block through lifecycle rules” is too vague; readiness eligibility must explicitly exclude or block that class before claim. |
| BLOCKING | Restart suppression | “Stale terminal rows must never suppress desired restart” conflicts with the need for suppression reasons and repeated-failure protection. The design must distinguish stale terminal state from active intentional suppression, otherwise it can restart-loop known-bad workers or ignore operator suppression. |
| BLOCKING | Multi-project boundaries | “Across configured projects” is undefined. The server needs a source of truth for configured projects, per-project auth/env/profile, cwd, branch/worktree policy, and failure isolation. Without that, one project’s stale state or lock repair could affect another project. |
| BLOCKING | Process model | The proposal cites `nohup` death and `setsid ... < /dev/null` survival, but does not specify the permanent launch contract: session, process group, stdio, signal handling, log capture, child reaping, and shutdown semantics. Worker liveness based on PID/PGID is unreliable unless the process tree contract is fixed. |
| BLOCKING | Failure taxonomy | The typed failure classes are listed but not tied to detection predicates, lifecycle transitions, restart policy, telemetry fields, or tests. For example, `readiness_decode_unavailable` could mean schema mismatch, transient provider output corruption, or code defect, each requiring different handling. |
| BLOCKING | Readiness schema | The specific mismatch `rewrite.acceptance` array vs `preClaimIntakePromptRewrite.Acceptance` string is only listed as an incident. The design does not say whether the contract should accept both formats, reject with a typed failure, version the schema, or repair the prompt/parser boundary. |
| BLOCKING | EMFILE | `resource_exhausted_emfile` is named, but the plan lacks a mitigation: file descriptor limits, bounded concurrency, handle closure audit, retry/backoff behavior, and doctor check criteria. A failure class without resource control will not prevent Snorri-style stoppage. |
| BLOCKING | Bead decomposition risk | This target is too large to decompose safely as-is. It spans supervisor storage, process launching, stale claim repair, readiness parsing, telemetry concurrency, doctor checks, and multi-project integration tests. Without explicit file/test boundaries, beads will overlap and agents will make incompatible state-machine choices. |
| WARNING | Code-defect beads | “File code-defect beads” from meta-scan is dangerous unless deduplicated and rate-limited with stable fingerprints. Otherwise repeated scans can spam duplicate beads for the same deterministic failure. |
| WARNING | Terminology | “terminal row,” “run-state,” “worker state,” “desired slot,” “live PID/PGID,” and “heartbeat” are not defined as separate records or fields. This invites inconsistent implementations. |
| WARNING | Doctor checks | “Add doctor checks” is too broad. The target should name exact checks: slot-store integrity, stale terminal suppression, PID/PGID/session validation, fd limit, JSONL lock/write health, project config reachability, and readiness schema compatibility. |
| WARNING | Agent dispatch | “May dispatch an agent only after deterministic repair fails” needs a strict stop rule. Otherwise the server can turn operational repair into autonomous code editing or bead mutation in violation of queue stewardship expectations. |
| NOTE | Good failure coverage | The observed incidents are concrete and useful, but they need to become normative state-machine rules rather than examples. |

### Verdict: BLOCK

### Summary

The proposal identifies the right failure surface, but it is not yet safe to decompose into beads. The main gaps are state ownership, lifecycle transition rules, stale/dead process proof, restart suppression semantics, and exact test/file boundaries. Tighten this into a state-machine spec with schema, predicates, transition table, and integration-test matrix before filing implementation beads.
