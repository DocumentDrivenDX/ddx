## Review: DDx Worker Self-Healing (revised design)

The extend-the-supervisor reframing resolves the r1 BLOCK (greenfield slot-store over a live supervisor). Remaining blockers are narrower and concrete — each verified against the target code.

### Findings

| Severity | Area | Finding |
|---|---|---|
| BLOCKING | Restart convergence | The design's claim *"Restart backoff remains the protection against rapid restart loops"* is false for the exact terminal class being expired. In `recordTerminalHistory`, `operator_attention`/`dirty_root` terminals are routed to `blockedTerminals` (`workers_supervisor.go:374-386`) and **never** appended to `restartEvents` (`:387-388`). `canStartMore` computes backoff *only* from `restartEvents` (`:420-438`). So once bead #1 makes blocks expire by `TerminalBlockTTL`, a worker that keeps re-hitting dirty-root/operator-attention/EMFILE will restart every TTL with **zero exponential backoff**. The plan must specify that expired-block restarts feed a throttle (either into `restartEvents` or a dedicated per-reason counter). |
| BLOCKING | JSONL streams | flock does not fix the "recurring merge conflicts" symptom the root cause names. `.ddx/metrics/locks.jsonl` is git-tracked and dirty right now (`git status: M`), and there is **no `.gitattributes` union-merge rule** for `*.jsonl`. A cross-process `flock` serializes same-host appends but does nothing for git merge conflicts across branches/worktrees — which is the *named* incident (root cause #4: "recurring merge conflicts for `.ddx/metrics/attempts.jsonl`, `.ddx/metrics/locks.jsonl`"). The design collapses two distinct problems (write interleaving vs. VCS merge) into one solution. Bead #4 needs a separate, explicit decision: union merge driver, per-writer file sharding, or move-out-of-git — distinct from the flock work. |
| BLOCKING | Review gate | The preserved-needs-review unblock predicate is undefined against how the state is actually stored. `preserved_needs_review` exists only as an execute-bead **event body string** (`"preserved-needs-review\ngate_summary=…"`, `execute_bead_loop.go:5487-5499`) — there is no durable bead status field and no defined "review-accepted/unblock" event kind or marker. Bead #5 says the bead becomes ready again when *"an operator records an explicit unblock/review-accepted note after the preserved event,"* but names no event kind/marker string, and this overlaps the LLM-level stale-blocker-precedence rule already in the intake prompt (`preclaim_intake_hook.go:667`). Bead #5 lists both `store.go` (deterministic) and `execute_bead_loop.go` without saying which layer owns the gate — two agents will implement incompatible markers/layers. |
| BLOCKING | Bead decomposition | No dependency/sequencing DAG is given for beads that mutate shared files and coupled state, violating the bead-authoring rubric's explicit-Deps requirement. Beads #1 and #2 both edit `workers_supervisor.go` (terminal-block struct + `canStartMore` vs. `snapshotWorkers` liveness predicate at `:318-319`); bead #6's EMFILE terminals are consumed by bead #1's suppression classifier (`isRestartBlockedWorker:524-542`); beads #5 and #6 both touch `cli/internal/agent`. Run in parallel worktrees these conflict, and the #1↔#6 coupling forces an ordering the plan never states — exactly the "incompatible state-machine choices" the review question targets. |
| WARNING | Restart suppression | Per-kind/profile scoping is not expressible in the data model, and the suppression *arithmetic* is unspecified. `WorkerDesiredState` carries a single `DefaultSpec` (`:42`), so all desired workers in a project share one profile/kind — there is nothing to scope by. The plan says "must not let one old terminal block suppress all desired workers" but never defines how many workers one active block suppresses when `DesiredCount > 1` (today: any block suppresses all, `:416-418`). |
| WARNING | Constants | `TerminalBlockTTL` is named with no value, no config field, and no stated relationship to `Restart.Backoff`/reconcile interval. The implementing agent will pick an arbitrary number. |
| WARNING | Liveness predicate | `workerRecordLive` lists "heartbeat is fresh when available" and "run-state not expired" with **no freshness thresholds**, the sidecar heartbeat's existence is unconfirmed (bead #2 hedges "`workerstatus/*` if needed"), and the plan doesn't state precedence between the new predicate and the existing `hasWorkerHandle` short-circuit (`:318`). |
| WARNING | JSONL bead scope | Bead #4 drops the exact-file precision the r1 trace established (`executions_mirror.go:339`, `worker_ingest.go:217`, `lockmetrics.go:120`, routing/attempt metrics), replacing it with "runtime metric/mirror writers under `cli/internal/agent` and `cli/internal/server`" — inviting incomplete coverage. |
| WARNING | Meta-scan | "clear dead-PID run-state" contradicts the design's own Non-Goal ("Do not treat PID-only liveness as sufficient when PGID/session evidence is available"). The meta-scan should reuse `workerRecordLive`, not raw PID checks. |
| WARNING | Resource recovery | Bead #6 leaves the cleanup **actor** and **thresholds** unspecified ("restarts with backoff only after cleanup/recheck" — worker or supervisor?), and "releases any claim it owns" requires file writes that can themselves fail under FD exhaustion; the ordering under EMFILE is not addressed. |
| NOTE | Ready beads | Beads #2 (PID+PGID liveness predicate) and #3 (string-or-`[]string` acceptance decoder) are execution-ready as written; #3 has a directly reusable pattern in `decodeReadinessAcceptanceList` (`preclaim_intake_hook.go:255-257`) already applied to `suggested_child_beads[].acceptance`. |

### Verdict: REQUEST_CHANGES

### Summary

The revised extend-not-replace framing is a real improvement and correctly retires the r1 greenfield BLOCK; four beads (notably #2 and #3) are close to execution-ready. But four blockers remain: (1) the promised restart backoff structurally does not apply to the blocked-terminal class being expired, so bead #1 can create a TTL-cadence restart loop; (2) flock does not address the git-merge-conflict symptom that root cause #4 explicitly names; (3) the preserved-needs-review "unblock" predicate has no defined event marker and straddles two ownership layers; and (4) no dependency DAG exists for beads that edit `workers_supervisor.go` and coupled terminal state concurrently. Resolve these four — plus the TTL value, suppression arithmetic, and per-file JSONL scope — before decomposition.
